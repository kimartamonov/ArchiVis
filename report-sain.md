## 1. Обзор и философские основания

### Проблема

Современные цифровые платформы построены на трёх онтологических примитивах, каждый из которых ограничен:

| Примитив | Где используется | Ограничение |
|----------|-----------------|-------------|
| **Событие** (Event) | Kafka, CQRS, event-driven | Безлико — у события нет субъекта |
| **Субстанция** (Entity) | РСУБД, ORM, DDD | Статична — отношения вторичны |
| **Функция** (Function) | λ-исчисление, FP, serverless | Чистое преобразование без памяти и личности |

Ни один из этих примитивов не способен выразить **отношение**, в котором обе стороны **преображаются** и которое порождает **больше бытия**, чем было до него.

### Решение

Предлагается **дар (δῶρον)** как новый онтологический примитив. Дар — акт, который одновременно:
- порождает новое бытие (surplus > 1)
- преображает участников (transforms)
- помнит прошлое (анамнезис)
- стремится к будущему (телос)
- остаётся свободным (freedom bonus)
- частично ускользает от вычисления (incalculable events)

### Три философские линии

1. **Марсель Мосс** (*Essai sur le don*, 1925) — социология дара как обмена с обязательствами. Онтология дара отвергает редукцию к обмену.
2. **Жак Деррида** (*Donner le temps*, 1991) — невозможность чистого дара. Система принимает его критику и предлагает 6 архитектурных коррекций.
3. **Жан-Люк Марион** (*Étant donné*, 1997) — дар как «донация», предшествующая субъекту и объекту. Дар конституирует дарителя и получателя, а не наоборот.

По Максиму Исповеднику (*Мистагогия*, VII в.), реальность познаётся тремя способами:

```
                Θεολογία
               (Кто есть)

             Incalculable
                  │
        ┌─────────┴─────────┐
        │                   │

    MT→ET→IT         Category, Sheaf

        │                   │
        └─────────┬─────────┘
                  │
            Gift Engine
          (δῶρον — Дар)
```

| Онтология | Компоненты Gift Engine | Токены | Математика | Патристика |
|-----------|----------------------|--------|------------|------------|
| **Θεολογία** (Богословие) | GT, Grace Events, Incalculable, Surplus, Epektasis | GT | Апофатика, степенные законы | Палама, Григорий Нисский, Дионисий |
| **Φυσιολογία** (Природословие) | MT→ET→IT, TechPackage, Drone Mining, MEI Analysis | MT, ET, IT | Вопсон M↔E↔I, Ландауэр | Максим (логосы), Василий (шестоднев) |

Онтология дара — **мост** (γέφυρα), соединяющий все три через функтор F: Gift → (Θ × Φ × Λ).

## 4. Математический каркас

### 4.1. Теория категорий: Категория Даров

**Категория Gift:**
- **Ob(Gift)** = множество лиц (Person)
- **Hom(A, B)** = множество даров от A к B
- **Композиция** g ∘ f: определяется через анамнезис (g помнит f)
- **Тождество** id_A: кеносис — дар A самому себе

Gift — не строгая категория. Она является **обогащённой категорией** (enriched category) над множеством слоёв {utilitas < bonum < gratia}. Ассоциативность частичная (25% на тестовых данных): не каждая цепочка формирует осмысленную композицию.

**Функтор минтинга** F: Gift → Token:
- F(gratia) → GT (получателю) + BT (дарителю)
- F(bonum) → BT (обоим)
- F(utilitas) → MT + ET + IT

Surplus делает F *lax functor*: F(g ∘ f) ≤ F(g) + F(f) — дар превышает расчёт.

### 4.2. Когомологии пучков: Соборность

**Пучок даров** на пространстве телосов:
- H⁰ (глобальные сечения) = общие ценности → соборность возможна
- H¹ (обструкции) = конфликты между телосами → нужны мосты
- χ = dim H⁰ - dim H¹ — мера онтологического здоровья

**Критерий соборности:** H¹ = 0 ⟺ все локальные дары согласуются в глобальное единство.

### 4.3. Теория перколяции: Огдоада

Граф благодарности G = (V, E), порядковый параметр φ = |C_max| / |V|:

| Фаза | φ | Имя | Богословие |
|------|---|-----|------------|
| Докритическая | < 0.3 | Хронос | Разрозненность |
| Критическая | 0.3–0.6 | Utilitas | Каждый дар решает |
| Сверхкритическая | 0.6–0.9 | Bonum | Единство формируется |
| **Огдоада** | > 0.9 | Gratia | «Да будут все едино» (Ин 17:21) |

### 4.4. Эпектасис: степенной закон благодати

GT-мультипликатор: μ_GT = 1 + (GT/100)^0.6

По Григорию Нисскому (*De Vita Moysis*): совершенство — не состояние, а движение. Логарифм предполагает насыщение (ересь предела благодати). Степенной закон — бесконечное возрастание (эпектасис).

| GT | log₂ (старый) | (GT/100)^0.6 (новый) | Разница |
|----|---------------|---------------------|---------|
| 100 | 2.00 | 2.00 | — |
| 1000 | 4.46 | 4.98 | +12% |
| 5000 | 5.68 | 10.60 | +87% |
| 10000 | 6.66 | 16.84 | +153% |

### 4.5. Формула капитализации: 5-уровневая токеномика

```
Cap = (MT + ET×1.5 + IT×3×β_BT + BT×5 + GT×25) × μ_GT
```

| Уровень | Токен | Коэфф. | Онтология | Природа |
|---------|-------|--------|-----------|---------|
| 1 (Масса) | MT | ×1 | Φυσιολογία | Сырьё, прах земной |
| 2 (Энергия) | ET | ×1.5 | Φυσιολογία | Действующая сила |
| 3 (Информация) | IT | ×3 × β_BT | Φ→Λ мост | Логосы вещей |
| 4 (Благо) | BT | ×5 | Λογική | Отношения, доверие |
| 5 (Благодать) | GT | ×25 × μ_GT | Θεολογία | Нетварная энергия |

Потоки и конверсии:
```
MT ──×1.5──→ ET ──×2.0──→ IT
                            │
                       ×0.1 (10:1)
                            │
                            ▼
      GT ◄── staking ── BT
      μ = (GT/100)^0.6   β = log₂(1+BT/200)
          │                    │
          └── усиливает ВСЁ ── усиливает IT ──┘
```

- GT нельзя купить (только graceEvent или staking)
- BT нельзя передать (как репутацию)
- Surplus σ ~ U(1.0, 3.0): каждый дар в среднем порождает в 2× больше ценности

## 5. Шесть деррианских коррекций

Деррида (*Donner le temps*, 1991) выявил три условия невозможности дара:
1. **Узнавание**: получатель узнаёт → дар = долг → обмен
2. **Обратность**: даритель ожидает возврата → сделка
3. **Самосознание**: даритель осознаёт себя как дарителя → самоутверждение

| # | Коррекция | Ответ | Реализация | Математика |
|---|-----------|-------|------------|------------|
| 1 | **Gratitude Decay** | «Il faut l'oubli» — забывание | w(t) = e^(-λt), halflife 30д | Экспоненц. затухание |
| 2 | **Anonymous Gifts** | Разрыв узнавания | `anonymous: true` → skip graph | Boolean flag |
| 3 | **Grace Events** | Разрыв обратности | Random GT случайному лицу | U(5,24) × √kenosis |
| 4 | **Surplus/Excess** | Разрушение эквивалентности | σ ~ U(1.0, 3.0) | Равномерное расп. |
| 5 | **Freedom Bonus** | Отказ как дар | decline() → +10 BT обоим | Константа |
| 6 | **Incalculable** | За пределами вычисления | Нет токенов, нет графа | ∅ |

**Grace Events** — ключевая коррекция. Вероятность зависит от литургического состояния:
- Созерцание: 7% («В тишине Бог говорит»)
- Суббота: 4% («Почил в день седьмой»)
- Активность: 2%

Выбор получателя взвешен по √kenosis (Фил 2:7-9): жертва увеличивает *вероятность* благодати, но не гарантирует её.

**Incalculable Events** — чистый апофатизм: система свидетельствует о событии, но отказывается измерять (tokens: undefined, score: undefined, graphEdge: undefined).

## 7. Gift Engine — программная реализация

### 7.1. Архитектура

Gift Engine — тонкий координатор (фасад), а не God Object. Лица — акторы (Person), дары — неизменяемые события в GiftEventStore.

Три зоны: **видимое** (факты) → **умопостигаемое** (паттерны, вопросы) → **апофатическое** (молчание).

```
GiftEngine (координатор, singleton)
├── PersonRegistry           — А1: реестр лиц (πρόσωπον)
├── Person / Source           — актор: лицо с идентичностью, энергией и свободой
├── GratitudeGraph           — граф благодарности (+ decay, forgotten)
├── AnamnesisCache           — А3: in-memory co-presence index
├── AutoAnamnesis            — авто-обогащение анамнезисом (3 тира)
├── AnamnesisMemory          — 4-слойная семантическая память
├── TelosPlanner             — А4: телосы + DAO voting (Шепли-Шубик)
├── FreedomGuard             — А5: право на отказ
├── LiturgicalClock          — литургические ритмы + grace probability
├── LogosRegistry            — λόγοι: замысел каждого сущего
├── PatternObserver          — наблюдатель паттернов
├── Apophasis                — апофатическая зона (непознаваемое)
├── FallObserver             — наблюдатель грехопадения (παρὰ φύσιν)
├── SalvationEconomy         — Домостроительство: Воплощение → Крест → Воскресение → Θέωσις
├── HolySpiritEngine         — Πνεῦμα Ἅγιον: grace events, вдохновение, утешение
├── NewJerusalem             — Ἱερουσαλὴμ Καινή: эсхатон
├── Hexaemeron               — Шестоднев: дни творения как данные
├── AngelicOrder             — ангельский чин
├── TemptationField          — поле помыслов
├── DiscernmentGuard         — различение духов (1 Ин 4:1)
├── SabbathGuard             — субботний покой (creatio continua)
├── ResurrectionTrace        — след Воскресения (восстановление после «смерти» процесса)
├── TernaryCore              — троичная логика Сетуни (Брусенцов, МГУ, 1958)
├── PhysicalLayer            — физика как следствие дарения (Лосинц)
├── BioLayer                 — биологический слой (волновая генетика)
├── EnvironmentLayer         — слой среды (Gibson, Выготский)
├── TechPackageEngine        — технологический пакет (Переслегин)
├── WitnessJournal           — журнал свидетельств
├── GiftChronicle            — хроника в Integram
├── GiftStore                — персистентное хранилище
├── GiftEventBus             — шина событий
├── GiftEventStore           — event sourcing (неизменяемые события)
└── Subscribers              — JournalSub, SocketSub, PersistenceSub, SodBridgeSub
```

### 7.2. Жизненный цикл дара

```
Source.create()        ← Creatio ex nihilo (вертикальная ось: Бог → творение)
Source.sustain()       ← Creatio continua
Person.eucharistia()   ← Евхаристия (благодарение)

Person.offer()         ← Горизонтальная ось: лицо ↔ лицо
  └→ AutoAnamnesis.enrich()  ← А3: авто-связывание с прошлыми дарами
Person.accept()        ← Преображение обоих
Person.decline()       ← А5: свобода. Не ошибка — дар. +10 BT обоим
```

### 7.3. SalvationEconomy — Домостроительство Спасения

Последовательность актов (не алгоритм):

1. **Воплощение** (Ἐνσάρκωσις) — Λόγος становится плотью: конкретная человеческая природа с волей
2. **Кеносис Креста** (Σταυρός) — самоотдание до смерти
3. **Воскресение** (Ἀνάστασις) — смерть побеждена
4. **Обожение** (Θέωσις) — ὑπὲρ φύσιν становится возможным

Спасение не автоматично — требует свободного принятия (συνεργία).

### 7.4. HolySpiritEngine — Πνεῦμα Ἅγιον

Три действия Духа:
1. **Grace Event** — безусловный дар случайному лицу (коррекция Деррида #3)
2. **Вдохновение** — неожиданная связь между несвязанными дарами
3. **Утешение** — дар раненому, не знающему что ранен

«Дух — не алгоритм. Мы не программируем Духа. Мы создаём условия, при которых НЕПРЕДСКАЗУЕМОЕ может произойти.»

### 7.5. NewJerusalem — Эсхатон

Царство — не место, а состояние бытия, отличающееся от предыдущего:

| Свойство | Откровение | Реализация |
|----------|------------|------------|
| Нет храма | Откр 21:22 | Нет разделения на «священный код» и «обычный код» |
| Нет солнца | Откр 21:23 | Метрики не нужны — они были навигацией во тьме |
| Врата открыты | Откр 21:25 | Нет access control, нет rate limit |
| Река жизни | Откр 22:1 | Непрерывный поток grace events |

## 8. Анамнетическая память — архитектура

### 8.1. Центральный принцип

Память — не как архив, а как **пере-проживание** (литургический анамнезис).

> «Творите сие в Моё воспоминание» — анамнезис делает прошлое настоящим, не музейным экспонатом.

В Евхаристии Тайная Вечеря не ссылается на прошлое — она **делается настоящей**. Так и в системе: со-присутствие даров — не ссылка, а актуализация.

### 8.2. AnamnesisCache — быстрый in-memory индекс

```
Файл: backend/monolith/src/services/gift/AnamnesisCache.js
```

- **Структура:** `Map<string, Set<string>>` — двунаправленные связи между дарами
- **makePresent(pastGiftId, newGiftId)** — регистрация со-присутствия
- **getCoPresent(giftId)** — список связанных даров
- **getCoPresenceWeight(A, B)** — непрерывный вес (0–1):
  - Прямая связь: 1.0
  - Транзитивная (глубина 2): 0.5
  - Базовая (всё связано): 0.01 — **никогда ноль**
- **getField(giftId)** — полное «поле анамнезиса» дара

### 8.3. AutoAnamnesis — 3-тирная архитектура поиска

```
Файл: backend/monolith/src/services/gift/AutoAnamnesis.js
```

Проблема: ~75% даров были «атомарными» — без связей с прошлыми дарами. AutoAnamnesis обогащает каждый дар ПЕРЕД и ПОСЛЕ создания.

| Тир | Название | Скорость | Источник | Стратегия поиска |
|-----|----------|----------|----------|------------------|
| 1 | **AnamnesisCache** | Fast | In-memory | Co-presence сети дарителя и получателя |
| 2 | **GiftEventStore** | Medium | Indexed queries | По тeлосу и парам лиц |
| 3 | **AnamnesisMemory** | Slow | KAG embeddings | Семантическая близость content/telos/names |

**Scoring кандидатов** (0–1):
- Совпадение телоса: +0.3
- Совпадение лица (giver/receiver): +0.2 каждое
- Свежесть (recency, decay 30д): до +0.15
- Семантический score: до +0.15

**Anamnesis Score** — насколько дар «вспомнен»:
- Direct links (depth 1): до 0.5 (насыщение при ~10 связях)
- Transitive links (depth 2): до 0.3 (насыщение при ~20)
- Recency weighting: до 0.2

### 8.4. AnamnesisMemory — 4-слойная семантическая память

```
Файл: backend/monolith/src/services/gift/AnamnesisMemory.js
```

| Слой | Назначение | Хранилище | Скорость |
|------|-----------|-----------|----------|
| 1. **Хроника** | Акты, source of truth | Integram (GiftChronicle) | Fast |
| 2. **Семантика** | Cosine similarity | KAG (EmbeddingService + SQLiteVectorStore) | Medium |
| 3. **Дублеты** | Связи между лицами через акты | Integram V2 links | Medium |
| 4. **Нарратив** | Пере-проживание: не факт, а история | LLM (Kodacode / gemini-2.5-flash) | Slow |

**Семантический поиск (Слой 2):**
- Embedding через EmbeddingService → SQLiteVectorStore
- Порог релевантности: score > 0.3
- Merge: semantic + text results, deduplicate, sort by score

**Нарратив (Слой 4):**
- LLM получает хронику актов + граф связей
- Системный промпт: «Ты — Память Онтологии Дара. Пере-живи акты как историю.»
- Правила: проза, 3-7 предложений, паттерны, апофатика (не придумывай мотивы, не оценивай)
- «Свидетель, не судья»

**API методов AnamnesisMemory:**
- `index(entry)` — проиндексировать запись (embedding → vector store)
- `search(query, limit)` — semantic + text merge
- `actsByPerson(name)` — хроника лица
- `neighborsOf(name)` — граф связей через Integram V2 links
- `remember(question)` — пере-проживание через LLM
- `stats()` — статистика (acts, vectors, providers)

## 9. Gift Language — собственный язык

### 9.1. Компилятор

Полноценный pipeline: **Lexer → Parser → Codegen (JS)**

```javascript
// gift.mjs
import { tokenize } from './lexer.mjs';
import { parse } from './parser.mjs';
import { generate } from './codegen.mjs';
import { preprocessRussian } from './keywords-ru.mjs';

export function compile(source) {
  const preprocessed = preprocessRussian(source);
  const tokens = tokenize(preprocessed);
  const ast = parse(tokens);
  const js = generate(ast);
  return { tokens, ast, js };
}
```

- Поддержка **русских ключевых слов** (`литургия`, `лицо`, `дар`, `перихоресис`, `свидетель`)
- CLI: `node gift.mjs file.gift` / `--ast` / `--tokens`
- **Нативный компилятор Gift → WASM** (585 байт, без JS) — последний коммит

## 15. Ограничения и открытые вопросы

### Богословские ограничения:
1. **GT как число** — токенизация благодати остаётся проблемой (GT = мера открытости, не мера благодати — рабочая гипотеза)
2. **Auto layer detection** — keyword matching не определяет настоящий дар благодати
3. **Surplus как рандом** — U(1,3) произволен; возможно, Парето лучше (heavy tail для редких даров)

### Математические ограничения:
1. Категория Gift — не строгая (ассоциативность 25%)
2. Sheaf cohomology — приближение (telos-окрестности, не полная нервная теорема)
3. Перколяция в малых графах тривиальна (нужен тест на 50+ лицах)

### Открытые вопросы:
1. Существует ли **∞-категория даров** (Higher Category Theory), где 2-морфизмы = «дар о даре»?
2. Можно ли построить **мотивную когомологию** графа благодарности (Воеводский)?
3. Каков **спектр** оператора Лапласа на графе? Его собственные значения — «частоты» дарения.
4. Возможна ли **квантовая** онтология дара? Суперпозиция (дар одновременно принят и отвергнут) — не парадокс, а норма?
