# 08_System_Context_and_Architecture.md: вопросы к заполнению

## Каких данных не хватает в Idea-Final

- Есть направление по стеку и принцип отделения engine от UI, но отсутствует целостная архитектурная форма системы.
- Не определены системный контекст, границы компонентов, место вычислений, способ развёртывания и интеграционные контуры.
- Не зафиксированы ключевые архитектурные решения, которые влияют на MVP и roadmap.

## Вопросы

- Где должен выполняться graph engine в MVP: в браузере, на отдельном backend-сервисе, в гибридной схеме?
- Нужен ли собственный backend ArchiLens или можно ограничиться frontend + прямое обращение к API Architeezy?
- Какие внешние системы входят в системный контекст кроме Architeezy: SSO, файловое хранилище, CI/CD, telemetry, demo datasets?
- Какой вариант развёртывания нужен для MVP: локальный запуск, статический web app, контейнер, self-hosted сервис?
- Почему для MVP выбран именно такой разрез слоев: connector, graph engine, insight engine, UI?
- Нужен ли API слой самого ArchiLens для внутреннего взаимодействия или только модульная библиотека?
- Как будет организовано управление состоянием между поиском, графом, фильтрами, анализом и экспортом?
- Какие вычисления должны быть синхронными, а какие асинхронными?
- Нужно ли выносить тяжелые расчеты в worker/background processing?
- Какой архитектурный контракт должен существовать между connector layer и graph engine?
- Нужен ли CLI/batch mode уже в MVP или это только архитектурный задел?
- Какой подход нужен для extension API: плагины на уровне интерфейсов TypeScript, runtime plugins, post-MVP?
- Как должна выглядеть высокоуровневая структура UI-модулей и их связи?
- Какие архитектурные решения являются необратимыми и требуют отдельной фиксации до начала реализации?
- Какие ограничения MVP должны влиять на архитектуру с самого начала: размер моделей, read-only режим, explainability, open source extensibility?
