# 01_Project_Brief.md: вопросы к заполнению

## Каких данных не хватает в Idea-Final

- Не определены заказчик, инициатор и контур принятия решений.
- Недостаточно явно сформулированы проверяемая гипотеза MVP и ожидаемый результат после MVP.
- Не зафиксированы стартовые ограничения проекта: команда, сроки, бюджет, формат поставки, критерии провала.

## Вопросы

- Кто инициатор проекта в практическом смысле: это конкурсная заявка, внутренний продукт, open source инициатива или партнерский showcase для Architeezy?
- Кто считается заказчиком документации и будущего MVP: вы, команда Architeezy, open source сообщество, конкретный пилотный клиент?
- Кто принимает продуктовые решения по проекту?
- Кто принимает архитектурные и технические решения?
- Какую главную гипотезу должен подтвердить MVP?
- Какой один результат MVP важнее всего: доказать полезность impact analysis, доказать ценность quality insights, показать силу API Architeezy, собрать основу для OSS-сообщества или получить пилотное внедрение?
- Как выглядит ожидаемый outcome после MVP: конкурсный демо-артефакт, рабочий инструмент для регулярного использования, старт open source проекта, база для коммерческого развития?
- Что будет считаться успехом MVP в измеримом виде?
- Что будет считаться неуспехом или причиной пересмотра идеи?
- Для какого горизонта планируется первая версия: демо на конкурс, 2-4 недели разработки, квартальный MVP, более длинная программа?
- Какие ограничения уже известны по срокам, людям, бюджету и доступности экспертов?
- Есть ли обязательные внешние ограничения: лицензии, корпоративные политики, требования Architeezy, ограничения API, требования конкурса?
- Нужно ли в brief сразу фиксировать географию или тип организаций, для которых продукт делается в первую очередь?
- Нужно ли позиционировать продукт как отдельный OSS-проект, модуль экосистемы Architeezy или reference implementation?
- Какой стартовый контекст нужно явно описать новому участнику команды, чтобы он правильно понял замысел без устных пояснений?
