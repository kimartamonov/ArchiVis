# 11_Acceptance_Criteria_and_Test_Strategy.md: вопросы к заполнению

## Каких данных не хватает в Idea-Final

- Критерии успеха сформулированы на уровне продукта, но не преобразованы в приемочные критерии по функциям.
- Не выбран главный сценарий приемки и не определены тестовые данные.
- Нет решения по глубине тестирования MVP и по приоритизации дефектов.

## Вопросы

- Какой один end-to-end сценарий будет главным сценарием приемки MVP?
- Какие acceptance criteria должны быть у подключения к Architeezy?
- Какие acceptance criteria должны быть у загрузки модели и построения графа?
- Какие acceptance criteria должны быть у Impact Analyzer?
- Какие acceptance criteria должны быть у coverage/hygiene анализа?
- Какие acceptance criteria должны быть у фильтров, поиска и навигации по графу?
- Какие acceptance criteria должны быть у экспортов?
- На каких тестовых данных будет происходить приемка: реальная модель, анонимизированная модель, synthetic demo dataset?
- Какие ожидаемые результаты должны быть заранее известны для проверки корректности метрик и impact-цепочек?
- Нужна ли ручная приемка со стороны архитектурного эксперта, и если да, в каком формате?
- Какие тесты обязательны для MVP: smoke, e2e, интеграционные, unit, визуальные, performance smoke?
- Какой уровень автоматизации тестов нужен в первой версии?
- Какие дефекты считаются блокирующими для релиза, а какие допустимы?
- Что входит в `definition of done` для capability slice и для всего MVP?
- Какие доказательства готовности должны быть собраны: demo video, checklist, sample exports, test report, known limitations?
