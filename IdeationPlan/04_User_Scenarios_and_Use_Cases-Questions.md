# 04_User_Scenarios_and_Use_Cases.md: вопросы к заполнению

## Каких данных не хватает в Idea-Final

- JTBD и продуктовые сценарии описаны на уровне намерения, но не разложены в пошаговые пользовательские потоки.
- Не определены предусловия, входные данные, ожидаемый результат и альтернативные ветки по каждому сценарию.
- Не выбран один канонический end-to-end сценарий MVP, по которому потом строить приемку и демо.

## Вопросы

- Какой один end-to-end сценарий должен стать главным сценарием MVP и основой demo script?
- С какого экрана и с какого триггера пользователь начинает каждый ключевой сценарий?
- Какие предусловия обязательны для начала сценария: подключенный Architeezy, выбранная модель, индексированный граф, известный элемент?
- Как пользователь находит нужный элемент: через поиск, список моделей, граф, таблицу, deeplink?
- Какие шаги пользователь проходит в сценарии `Impact Analysis` от выбора элемента до получения результата?
- Какие именно выходы должны появляться по завершении `Impact Analysis`: список affected elements, слои, диаграммы, risk card, export?
- Какой шаг за шагом сценарий у `Model Coverage & Hygiene`?
- Нужен ли отдельный пошаговый сценарий для `Criticality Radar` уже в MVP или это post-MVP use case?
- Как выглядит сценарий подготовки evidence-based narrative для стейкхолдеров?
- Какие сценарии должны поддерживать экспорт, а какие только просмотр внутри UI?
- Какие пользовательские ошибки и отклонения нужно заранее описать: элемент не найден, данных недостаточно, API не отвечает, модель слишком большая, граф не строится?
- Что считается успешным завершением каждого сценария в проверяемом виде?
- Какие сценарии являются основными, какие второстепенными, а какие только stretch?
- Какие сценарии должны поддерживать работу на реальной модели, а какие можно показывать только на demo dataset?
- Какие сценарии нужно явно исключить из MVP, чтобы они не маскировались под "почти готовые"?
