# 08_System_Context_and_Architecture: ответы и принятые решения

## Архитектурный принцип

Главное архитектурное решение для ArchiLens OSS уже намечено в `Idea-Final` и его нужно зафиксировать окончательно: **анализ должен быть отделен от визуализации**. Но этого мало. Нужно выбрать еще и практическую форму MVP: где выполняются вычисления, нужен ли отдельный backend, как идет поток данных от Architeezy к UI и какие решения считаются необратимыми.

Для первой версии оптимальна архитектура, которая минимизирует инфраструктурную сложность, но не запирает продукт в purely-front-only схему.

## Ответы на вопросы

### Где должен выполняться graph engine в MVP

Для MVP лучше выбрать **гибридную схему с выполнением graph engine в приложении ArchiLens, но не привязывать его жестко к main UI thread браузера**.

Практически это означает:

- нормализация модели и базовые расчеты могут выполняться в клиентском приложении;
- тяжелые операции должны быть готовы к выносу в `Web Worker`;
- архитектурно engine должен быть headless-модулем, который можно позже запускать и на backend, и в CLI.

Почему не чисто backend-only:

- это увеличит сложность MVP;
- потребует отдельного server-side orchestration уже на старте;
- не обязательно для моделей малого и среднего размера.

Почему не жестко “только в браузере”:

- это создаст лишний архитектурный тупик для CLI, batch и роста нагрузки.

### Нужен ли собственный backend ArchiLens

Да, для MVP лучше предусмотреть **тонкий backend/BFF слой**, а не делать чистый frontend с прямым доступом к API Architeezy.

Почему это лучше:

- безопаснее для токенов и конфиденциальных данных;
- упрощает работу с конфигурацией подключения;
- дает единое место для интеграционного и диагностического слоя;
- позволяет позже вынести более тяжелые вычисления без смены публичного контракта.

Но этот backend должен быть минималистичным:

- проксирование и нормализованный доступ к данным;
- конфигурация подключения;
- возможно, кэш и служебная диагностика.

Не нужно в MVP строить тяжелую распределенную серверную платформу.

### Какие внешние системы входят в системный контекст кроме Architeezy

Для MVP системный контекст включает:

- `Architeezy API` как основной источник данных;
- `demo datasets` как управляемый источник для тестов и демонстрации;
- `локальное или self-hosted окружение запуска`;
- `CI/CD` для сборки и проверки OSS-проекта;
- `логирование/диагностика` внутри приложения.

Что не нужно считать обязательной частью MVP-контекста:

- SSO как отдельную интеграцию;
- внешнюю telemetry SaaS;
- файловое хранилище как самостоятельную платформенную зависимость;
- внешние аналитические сервисы.

### Какой вариант развёртывания нужен для MVP

Лучший вариант для MVP:

> `self-hosted контейнеризуемое веб-приложение`, которое можно запускать локально или в контролируемом окружении команды.

Это означает:

- локальный запуск для разработки;
- контейнерный запуск для демонстрации и пилотного использования;
- отсутствие зависимости от внешнего облачного сервиса как обязательного режима.

Почему это лучше статического web app:

- нужен тонкий backend;
- безопасность и конфигурация подключения упрощаются;
- лучше соответствует требованию работы с конфиденциальными моделями.

### Почему для MVP выбран именно такой разрез слоев: connector, graph engine, insight engine, UI

Этот разрез слоев — не академическая красота, а прямой ответ на продуктовые риски.

- `Connector Layer` изолирует нестабильность и особенности Architeezy API.
- `Graph Engine` отвечает за нормализацию source data в единую аналитическую модель.
- `Insight Engine` отвечает за метрики, impact analysis и quality logic.
- `UI` отвечает за навигацию, интерпретацию и экспорт результата.

Если смешать эти слои:

- любая смена API начнет ломать UI;
- логика анализа станет нетестируемой;
- расширяемость превратится в миф;
- explainability будет трудно поддерживать.

### Нужен ли API слой самого ArchiLens

Да, но в MVP это должен быть **внутренний application API**, а не публичная внешняя платформа.

То есть нужен:

- BFF/API слой между UI и внутренними модулями;
- устойчивый контракт на операции `load model`, `get elements`, `run impact analysis`, `get coverage`, `export`.

Не нужен на старте:

- полноценный публичный REST/GraphQL API для сторонних интеграций.

### Как будет организовано управление состоянием между поиском, графом, фильтрами, анализом и экспортом

Нужно разделить состояние на несколько уровней:

- `application state`: источник, выбранная модель, статусы загрузки;
- `navigation state`: текущий экран, выбранный элемент;
- `analysis state`: глубина анализа, результаты impact, coverage summary;
- `view state`: фильтры, сортировка, режим представления;
- `export state`: формируемый артефакт и его статус.

Практически это хорошо ложится на централизованный state container вроде `Zustand`, но с модульным разделением store по зонам ответственности.

Главное правило:

> derived UI state не должен дублировать source analysis data без необходимости.

### Какие вычисления должны быть синхронными, а какие асинхронными

Синхронными можно оставить только очень легкие операции:

- локальная фильтрация уже загруженного списка;
- выбор элемента;
- переключение представления результата.

Асинхронными должны быть:

- загрузка модели;
- нормализация и индексирование;
- расчет impact analysis;
- подготовка экспортов;
- любые потенциально тяжелые пересчеты метрик.

Это важно и для UX, и для архитектуры надежности.

### Нужно ли выносить тяжелые расчеты в worker/background processing

Да, это нужно **заложить сразу**, даже если первая версия будет использовать это только частично.

Практическое решение:

- базово поддержать запуск тяжелых вычислений вне main UI thread;
- для frontend-выполнения использовать `Web Worker` как основной механизм;
- если позже появится server-side execution, контракт движка должен остаться тем же.

### Какой архитектурный контракт должен существовать между connector layer и graph engine

Контракт должен быть основан не на сырых ответах Architeezy, а на **нормализованном ingress-модуле**.

Connector должен отдавать:

- сырой, но структурированный набор моделей, элементов, отношений и view;
- явные сведения о проблемах загрузки;
- стабильный формат входных данных для graph engine.

Graph engine должен ожидать:

- четко типизированный input;
- отсутствие UI-зависимостей;
- возможность работать в pure/headless режиме.

То есть connector знает внешнюю систему, а graph engine знает только нормализованный контракт.

### Нужен ли CLI/batch mode уже в MVP

Как готовый пользовательский режим — нет. Как архитектурный задел — да, обязательно.

Это значит:

- engine не должен зависеть от React/UI;
- core analysis должен быть запускаемым из headless-контекста;
- экспорт и вычисление метрик должны быть отделены от браузерного представления.

Сам CLI можно оставить post-MVP.

### Какой подход нужен для extension API

Для MVP нужен **design-for-extension, но не runtime plugin system**.

Лучший подход:

- расширяемость на уровне TypeScript-интерфейсов и внутренних extension points;
- модульные контракты для метрик, quality rules и exporters;
- отсутствие runtime plugin loading в первой версии.

Иначе говоря, extension API в MVP — это архитектурная дисциплина, а не продуктовая функция.

### Как должна выглядеть высокоуровневая структура UI-модулей и их связи

Высокоуровневая структура UI для MVP:

- `App Shell`;
- `Source / Model Selection`;
- `Model Explorer`;
- `Element Table`;
- `Global Graph View`;
- `Element Detail Panel`;
- `Impact Analysis Panel`;
- `Coverage & Quality View`;
- `Export Actions`.

Связи между ними должны быть такими:

- выбор модели задает общий контекст;
- поиск и таблица выбирают элемент;
- граф и карточка элемента отражают тот же selection state;
- impact panel использует выбранный элемент и параметры анализа;
- coverage view может переводить пользователя обратно к элементу и impact;
- export actions доступны из результата анализа и coverage lists.

### Какие архитектурные решения являются необратимыми

До начала реализации нужно отдельно зафиксировать следующие необратимые или дорогие для смены решения:

- наличие собственного backend/BFF или его отсутствие;
- каноническая доменная модель и граница source vs derived сущностей;
- разделение `connector / graph engine / insight engine / UI`;
- место выполнения тяжелых вычислений и модель их асинхронности;
- правило read-only интеграции;
- формат внутреннего контракта между слоями;
- стратегия обращения с конфиденциальными данными.

Ошибиться здесь опаснее, чем ошибиться в конкретном экране.

### Какие ограничения MVP должны влиять на архитектуру с самого начала

На архитектуру сразу должны влиять следующие ограничения:

- `одна модель за раз`;
- `read-only режим`;
- `desktop-first UX`;
- `объяснимость метрик`;
- `работа с неполными данными`;
- `self-hosted / controlled deployment`;
- `будущая OSS-расширяемость`;
- `целевой размер модели малого и среднего масштаба`.

То есть архитектура не должна обещать все, но должна честно обслуживать именно этот профиль использования.

## Итоговая архитектурная форма MVP

MVP ArchiLens OSS следует строить как self-hosted веб-приложение с тонким backend/BFF, который безопасно работает с Architeezy API, и headless аналитическим ядром, отделенным от UI. Поток данных должен идти через `connector -> graph engine -> insight engine -> UI`, а тяжелые вычисления должны выполняться асинхронно и быть готовы к выносу в worker или server-side execution. Такая схема минимально сложна для первой версии, но не запирает продукт в тупиковую front-only архитектуру и хорошо поддерживает будущие CLI, batch и extension сценарии.
