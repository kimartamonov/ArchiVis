# 11_Acceptance_Criteria_and_Test_Strategy: ответы и принятые решения

## Принцип приемки MVP

Для ArchiLens OSS приемка не должна быть расплывчатой. Продукт либо позволяет пройти основной рабочий сценарий на реальной модели с понятным результатом, либо нет. Поэтому стратегия приемки должна опираться на один канонический e2e-flow и дополняться проверками устойчивости, корректности вычислений и доверия к результату.

## Ответы на вопросы

### Какой один end-to-end сценарий будет главным сценарием приемки MVP

Главный сценарий приемки:

> Пользователь подключается к Architeezy, выбирает модель, находит элемент, запускает impact analysis на выбранную глубину, получает затронутые элементы, слои и диаграммы, а затем экспортирует результат.

Если этот сценарий не проходит стабильно, MVP не принят.

### Какие acceptance criteria должны быть у подключения к Architeezy

Критерии приемки подключения:

- система может установить соединение с настроенным источником;
- пользователь видит понятный статус подключения;
- при ошибке авторизации или недоступности API система показывает диагностируемую ошибку;
- секреты не требуют ручного ввода в unsafe client-side flow;
- после успешного подключения пользователь может получить список доступных моделей.

### Какие acceptance criteria должны быть у загрузки модели и построения графа

Критерии приемки загрузки и graph build:

- система загружает одну выбранную модель из Architeezy;
- из загруженных данных строится единый канонический граф;
- элементы, отношения и диаграммы связаны корректно там, где данные это позволяют;
- проблемы данных фиксируются как warnings, а не приводят к silent failure;
- после загрузки доступны поиск, таблица и запуск impact analysis;
- время загрузки и подготовки модели укладывается в целевой диапазон для тестовой модели MVP.

### Какие acceptance criteria должны быть у Impact Analyzer

Критерии приемки Impact Analyzer:

- пользователь может выбрать элемент из UI;
- пользователь может выбрать глубину анализа 1, 2 или 3;
- система возвращает список затронутых элементов;
- система показывает затронутые слои;
- система показывает затронутые диаграммы;
- результат можно просмотреть как минимум в табличном и подграфовом виде;
- пользователь понимает, почему элемент или связь попали в результат;
- результат формируется в пределах целевой производительности для test dataset'а.

### Какие acceptance criteria должны быть у coverage/hygiene анализа

Критерии приемки coverage/hygiene:

- система отображает orphan elements;
- система отображает low-coverage elements;
- пользователь может перейти от проблемного элемента к его карточке или impact analysis;
- пользователь может выгрузить список проблемных элементов в `CSV`;
- при неполных данных система явно показывает ограничения анализа.

### Какие acceptance criteria должны быть у фильтров, поиска и навигации по графу

Критерии приемки:

- поиск по имени находит ожидаемый элемент в тестовом наборе;
- таблица поддерживает фильтрацию по типу и слою;
- пользователь может перейти из таблицы или результатов поиска к элементу;
- пользователь может переключиться к связанному элементу из impact result;
- графовый режим не является единственным способом доступа к результату;
- интерактивные операции не приводят к заметной деградации UX на целевой модели.

### Какие acceptance criteria должны быть у экспортов

Критерии приемки экспортов:

- `CSV` экспортируется без ошибки для impact result и coverage list;
- `GraphML` экспортируется для выбранного подграфа impact analysis;
- в экспорт попадают обязательные поля и идентификаторы;
- экспортируемый файл можно открыть внешним инструментом или проверить структурно;
- пользователь понимает, какой именно результат он экспортировал.

### На каких тестовых данных будет происходить приемка

Приемка должна идти на трех типах данных:

- `synthetic happy-path dataset`;
- `анонимизированная или безопасная реальная модель`;
- `broken-data dataset`.

Почему все три важны:

- первый нужен для детерминированной проверки;
- второй подтверждает реальную полезность;
- третий проверяет устойчивость.

### Какие ожидаемые результаты должны быть заранее известны

Нужно заранее зафиксировать known expected results хотя бы для нескольких сценариев:

- для выбранного seed-элемента известен expected set затронутых элементов на глубину 1 и 2;
- известен expected список диаграмм, в которых присутствует элемент;
- известны элементы, которые должны быть orphan;
- известны элементы, которые должны попасть в low-coverage.

Без заранее известного expected output невозможно различить “работает” и “кажется правдоподобным”.

### Нужна ли ручная приемка со стороны архитектурного эксперта

Да, ручная экспертная приемка нужна обязательно.

Формат:

- walkthrough главного e2e-сценария;
- эксперт проверяет, что результат impact analysis выглядит правдоподобно и полезно;
- эксперт оценивает объяснимость метрик и результата;
- замечания фиксируются как product feedback, а не только как баги.

Это особенно важно, потому что у такого продукта недостаточно лишь машинной корректности — нужна еще и профессиональная правдоподобность.

### Какие тесты обязательны для MVP

Обязательные тесты MVP:

- `smoke tests` запуска и загрузки приложения;
- `unit tests` для graph normalization и базовых аналитических функций;
- `integration tests` для connector layer и преобразования входных данных;
- `e2e test` для главного сценария impact analysis;
- `performance smoke` на целевой модели;
- ручная экспертная приемка.

Необязательные для MVP, но желательные позже:

- визуальные regression tests на графовые представления;
- большой набор snapshot tests UI;
- полноценное performance benchmarking framework.

### Какой уровень автоматизации тестов нужен в первой версии

Для первой версии нужен **умеренный, но умный уровень автоматизации**.

Что автоматизировать обязательно:

- core normalization logic;
- core metric calculations;
- основной e2e-flow;
- экспортные форматы на уровне структуры.

Что можно оставить в основном на ручную проверку:

- тонкие UX-аспекты графового режима;
- экспертную оценку правдоподобности аналитики.

### Какие дефекты считаются блокирующими для релиза

Блокирующие дефекты:

- невозможно подключиться к источнику или загрузить модель;
- граф не строится или строится некорректно для тестового случая;
- impact analysis падает, зависает или выдает явно неверный результат на known dataset;
- экспорт не работает;
- UI делает невозможным прохождение основного сценария;
- ошибки данных не диагностируются и приводят к silent corruption результата;
- утечка или небезопасная обработка чувствительных данных.

Допустимые дефекты для MVP:

- шероховатости визуализации;
- неидеальная работа второстепенных фильтров;
- ограниченная полировка non-critical экранов;
- best-effort поддержка нецелевых браузеров.

### Что входит в `definition of done` для capability slice и для всего MVP

`Definition of done` для capability slice:

- функциональность реализована;
- покрыта релевантными тестами;
- обработаны основные ошибки;
- задокументированы ограничения;
- capability встроен в главный пользовательский поток, если влияет на него.

`Definition of done` для MVP:

- проходит главный e2e-сценарий;
- выполнены acceptance criteria по загрузке, анализу, coverage и экспорту;
- есть sample datasets;
- есть manual expert validation;
- есть runbook, README и known limitations;
- производительность и устойчивость укладываются в agreed baseline.

### Какие доказательства готовности должны быть собраны

Для доказательства готовности MVP нужно собрать:

- checklist приемки;
- sample exports `CSV` и `GraphML`;
- краткий test report;
- перечень known limitations;
- demo script;
- по возможности короткое demo video или screen capture;
- результаты экспертной валидации на реальной или анонимизированной модели.

## Итоговая стратегия приемки MVP

MVP ArchiLens OSS принимается не по числу реализованных экранов, а по способности стабильно пройти один канонический сценарий impact analysis на реальной модели с корректной загрузкой данных, объяснимым результатом и рабочим экспортом. Автоматизация должна покрывать graph normalization, core analysis и основной e2e-flow, а финальная приемка обязательно дополняется ручной валидацией со стороны архитектурного эксперта и набором явных доказательств готовности.
