# 15_Issue_Decomposition_Rules: ответы и принятые решения

## Принцип декомпозиции backlog

После capability slices самая большая опасность — снова разрезать работу по техническим слоям и потерять вертикальную ценность. Поэтому правила декомпозиции должны удерживать две вещи одновременно:

- каждый milestone должен собираться из задач, которые реально приближают нас к рабочему пользовательскому результату;
- каждая отдельная задача должна быть достаточно маленькой, чтобы ее можно было оценить, проверить и завершить без скрытого хвоста.

## Ответы на вопросы

### Какую структуру backlog нужно использовать

Оптимальная структура:

> `milestones -> capability slices -> issues`

Не стоит использовать только плоский список issues с label'ами. Для MVP ArchiLens OSS слишком важно удерживать связь между roadmap, slices и конкретной реализацией.

Практически это выглядит так:

- `milestone` задает этап поставки;
- `capability slice` задает вертикальный пользовательский результат;
- `issue` — атомарная задача, которая продвигает slice.

### Какие типы issues нужны в проекте

Для проекта достаточно следующих типов issues:

- `feature`;
- `integration`;
- `analysis`;
- `ui`;
- `export`;
- `spike`;
- `decision`;
- `validation`;
- `docs`;
- `bug`;
- `refactor`.

Почему полезно выделить `analysis` отдельно:

- у продукта есть специфический слой graph/impact logic, который не сводится ни к UI, ни к integration.

### Какие поля должны быть обязательными у каждого issue

Обязательные поля issue:

- `Title`;
- `Type`;
- `Milestone`;
- `Capability Slice`;
- `Why / user outcome`;
- `Scope`;
- `Out of scope`;
- `Dependencies`;
- `Acceptance criteria`;
- `Definition of done hints`;
- `Owner` или роль-владелец.

Если issue не может ответить хотя бы на `why`, `scope` и `acceptance`, он еще не готов к реализации.

### Какой шаблон issue нужен для команды и/или AI-генерации

Рекомендуемый шаблон issue:

- `Контекст`: зачем существует эта задача.
- `Пользовательский или технический результат`: что станет возможным.
- `Scope`: что входит.
- `Out of scope`: что сознательно не входит.
- `Dependencies`: что должно быть готово раньше.
- `Acceptance criteria`: как проверим готовность.
- `Artifacts`: какие тесты, экспорты, заметки или документы ожидаются на выходе.

Для AI-генерации этот шаблон особенно важен: без него модель будет производить длинные, но размытые задачи.

### Какие правила именования issues и milestones нужно зафиксировать

Правила именования milestones:

- кратко и по результату, например `M1 Valuable MVP`, `M2 Usable MVP`.

Правила именования issues:

- начинать с глагола действия;
- отражать результат, а не технический слой;
- по возможности указывать slice.

Хорошо:

- `Load selected model into canonical graph`;
- `Run impact analysis for selected element`;
- `Export impact result to GraphML`.

Плохо:

- `Graph stuff`;
- `Backend work`;
- `UI fixes`.

### Какой уровень гранулярности считается правильным для одного issue

Правильный issue должен быть:

- достаточно маленьким, чтобы иметь понятный конец;
- достаточно крупным, чтобы давать наблюдаемый результат;
- достаточно узким, чтобы acceptance criteria были короткими и конкретными.

Хорошее эвристическое правило:

> Один issue должен либо закрывать один маленький capability шаг, либо один четко ограниченный технический блок, без которого capability не может заработать.

### Как понять, что issue слишком большой и требует split

Issue слишком большой, если:

- его acceptance criteria похожи на mini-specification;
- он требует нескольких независимых видов тестов и артефактов;
- в нем смешаны integration, analysis logic и UI без единого результата;
- его нельзя завершить без цепочки из 3-5 подзадач;
- непонятно, что можно показать после его завершения.

### В каких случаях split, наоборот, не нужен и только вреден

Split вреден, если:

- он разрывает один естественный рабочий шаг на бессмысленные микро-issues;
- после разбиения отдельные задачи теряют проверяемый смысл;
- появляется много “сначала данные, потом кнопка, потом обработчик, потом сообщение”, хотя это один маленький capability;
- coordination overhead становится больше, чем сама работа.

То есть нельзя доводить декомпозицию до уровня “issue на каждую строку архитектуры”.

### Какие зависимости между issues допустимо фиксировать и в каком виде

Допустимы только meaningful dependencies:

- `blocks` — эта задача блокирует следующую;
- `depends on` — этой задаче нужно завершение другой;
- `validates` — задача проверяет ранее принятое решение или slice;
- `follows from` — задача логически продолжает предыдущий capability step.

Не нужно строить гигантский dependency graph ради самого graph'а. Связи должны оставаться читаемыми.

### Что входит в `definition of ready` для issue

Issue готов к работе, если:

- он привязан к milestone и capability slice;
- понятно, зачем он нужен;
- scope и out of scope явно описаны;
- зависимости либо закрыты, либо известны;
- acceptance criteria сформулированы;
- понятен expected artifact или observable result.

Если этого нет, задача не ready, даже если “примерно понятно, что делать”.

### Что входит в `definition of done` для issue

Issue считается done, если:

- реализован заявленный результат;
- acceptance criteria выполнены;
- добавлены нужные тесты или validation evidence;
- задокументированы ограничения, если они есть;
- нет скрытого хвоста, без которого результат на практике не работает.

Для MVP важно не закрывать issue формально, если capability все еще нельзя реально использовать.

### Какие канонические паттерны декомпозиции нужны для типовых slices

**Для integration issues**:

- контракт источника;
- загрузка ресурса;
- нормализация;
- обработка ошибок;
- тест на known dataset.

**Для graph engine issues**:

- входная структура;
- преобразование в canonical model;
- инварианты;
- тесты ожидаемого результата.

**Для UI flow issues**:

- entry point;
- пользовательское действие;
- отображение результата;
- ошибка и пустое состояние.

**Для export issues**:

- структура формата;
- генерация;
- sample artifact;
- проверка открываемости или структурной валидности.

**Для testing/validation issues**:

- dataset;
- expected outcome;
- автоматическая или ручная проверка;
- evidence output.

**Для docs issues**:

- для кого документ;
- какой сценарий он поддерживает;
- какие ограничения и prerequisites описывает.

### Как нарезать decision/spike/validation issues отдельно от implementation issues

Эти типы задач нужно выделять отдельно и не смешивать с implementation.

`decision issue`:

- цель — выбрать вариант;
- выход — принятое решение и rationale.

`spike issue`:

- цель — уменьшить неопределенность;
- выход — короткий прототип, измерение или вывод.

`validation issue`:

- цель — подтвердить пользу или корректность;
- выход — evidence, expected vs actual, expert feedback.

Главное правило:

> spike не должен маскироваться под feature, а feature не должен скрывать в себе нерешенный spike.

### По каким правилам собирать milestone из issues

Milestone должен собираться не по типам задач, а по готовности capability result.

В milestone должны входить:

- несколько implementation issues;
- нужные decision/spike tasks, если без них capability нельзя безопасно реализовать;
- хотя бы одна validation задача;
- необходимые docs/issues, если без них результат нельзя использовать.

Milestone нельзя считать завершенным, если закрыт код, но не закрыта валидация или доказательство готовности.

### Какие антипаттерны backlog нужно явно запретить

Нужно явно запретить:

- issues без acceptance criteria;
- issues без связи с milestone и slice;
- слоевые mega-issues вроде `сделать backend`;
- скрытые epics, замаскированные под один issue;
- смешение spike и implementation в одной задаче;
- задачи “улучшить UX” без конкретного результата;
- backlog, где красивые post-MVP фичи стоят рядом с P0 как равноправные.

### Если issues генерируются автоматически, какие требования должны быть к AI-промпту, входным артефактам и проверке результата человеком

Если использовать AI-генерацию issues, нужны жесткие правила.

AI prompt должен включать:

- `Idea-Final`;
- зафиксированный `P0` scope;
- capability slice map;
- decision backlog;
- acceptance strategy;
- правила issue decomposition.

AI должен генерировать:

- только задачи внутри заданного milestone/slice;
- задачи с явным `scope / out of scope / acceptance`;
- задачи, не выходящие за границы MVP.

Проверка человеком обязательна для каждого пакета AI-generated issues:

- удаление дубликатов;
- вынос слишком больших задач в split;
- возврат задач без наблюдаемого результата;
- проверка, что AI не протащил hidden scope creep.

## Итоговые правила декомпозиции

Backlog ArchiLens OSS должен строиться по схеме `milestone -> capability slice -> issue`, где каждый issue имеет четкий результат, границы и проверяемые критерии готовности. Декомпозиция должна усиливать один сквозной пользовательский поток, а не раскладывать работу по технологическим башням. Все нерешенные вопросы должны жить в decision/spike/validation issues отдельно, а любые автоматически сгенерированные задачи обязаны проходить ручную редактуру до того, как попадут в рабочий backlog.
