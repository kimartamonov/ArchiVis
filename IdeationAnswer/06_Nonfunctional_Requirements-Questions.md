# 06_Nonfunctional_Requirements: ответы и принятые решения

## Принцип нефункциональной рамки

Для ArchiLens OSS нефункциональные требования не вторичны. Если impact analysis работает медленно, непредсказуемо или выглядит как недостоверная “магия”, продукт не будет использоваться независимо от того, насколько правильно он реализован функционально.

Поэтому для MVP нужно зафиксировать не идеальные enterprise-показатели, а реалистичные и проверяемые пределы качества первой версии.

## Ответы на вопросы

### Какой размер модели должен считаться целевым для MVP

Для MVP разумно зафиксировать целевой диапазон, а не обещать бесконечный масштаб.

Целевой размер модели для первой версии:

- до `5 000` элементов;
- до `15 000` отношений;
- до `300` диаграмм.

Это хороший компромисс:

- уже достаточно реалистично для ощутимой архитектурной боли;
- еще не заставляет оптимизировать продукт как high-scale graph platform с первого дня.

Важно: это не жесткий технический предел, а **целевой размер, на котором MVP должен чувствовать себя уверенно**.

### За какое время допустимо загрузить модель и построить граф в целевом сценарии

Для целевого сценария допустимый ориентир такой:

- загрузка и подготовка модели малого/среднего размера: до `10-15 секунд`;
- на хорошем локальном или офисном окружении желательно ближе к `5-8 секундам`.

Дольше можно терпеть только в редких пограничных случаях. Если пользователь регулярно ждет заметно больше, продукт начнет восприниматься как тяжеловесный и “не для оперативной работы”.

### За какое время пользователь должен получать результат impact analysis после выбора элемента

Целевой SLA для MVP:

- типичный impact analysis: до `2 секунд`;
- допустимый максимум в сложном случае: до `5 секунд`.

Почему это важно:

- impact analysis — главная операция продукта;
- если результат идет слишком долго, пользователь возвращается к ручному анализу или интуитивному решению.

### Какая задержка допустима для фильтрации, поиска и навигации по графу

Для интерактивных действий нужны более строгие ориентиры:

- поиск и фильтрация списка: визуальный отклик до `300 мс`, допустимый максимум до `1 секунды`;
- переход к элементу и открытие карточки: до `1 секунды`;
- простая навигация по графу и переключение представлений: без заметных подвисаний при целевом размере модели.

Идея проста: основной поток должен ощущаться отзывчивым, даже если тяжелые вычисления происходят чуть дольше.

### Какие браузеры и платформы должны официально поддерживаться в MVP

Для MVP достаточно официальной поддержки:

- `Chrome` последних версий;
- `Edge` последних версий.

Желательно, но не критично:

- `Firefox`.

`Safari` можно считать `best effort`, но не основным таргетом первой версии, если нет специальных требований.

### Нужна ли адаптация под мобильные устройства или достаточно desktop-only режима

Для первой версии достаточно **desktop-only режима**.

Почему это не слабость, а осознанное решение:

- сценарии impact analysis и графовой навигации по природе плохо работают на мобильном экране;
- мобильная адаптация съест время, не усиливая core value.

Если нужно, можно сделать базовую деградацию верстки без официальной поддержки сложного mobile UX.

### Какой уровень устойчивости обязателен

Для MVP обязателен принцип **graceful degradation**.

Это означает:

- при временной недоступности API система должна явно показывать ошибку подключения и предлагать повторить попытку;
- при частичной ошибке загрузки система должна по возможности завершать импорт доступных данных и помечать неполноту;
- при сбое построения графа система должна давать диагностическое сообщение, а не зависать без объяснения;
- UI не должен ломаться полностью из-за проблем на одном элементе или одной диаграмме.

Иными словами, у продукта нет права быть “хрупким”.

### Требуется ли кэширование данных, и если да, то какие ограничения по актуальности кэша допустимы

Да, **кэширование желательно уже в MVP**, но в ограниченном и прозрачном виде.

Что кэшировать:

- загруженную модель текущей сессии;
- результаты индексации;
- предрассчитанные базовые метрики.

Ограничение по актуальности:

- в рамках одной сессии кэш допустим;
- между сессиями лучше считать кэш потенциально устаревшим, если нет явного механизма refresh/versioning.

Правильная позиция:

> В MVP допустим session-level cache, но нельзя скрывать от пользователя потенциальную неактуальность данных.

### Какие требования по безопасности уже известны

Даже для MVP нужно зафиксировать базовые security requirements:

- продукт должен работать с архитектурными данными как с потенциально конфиденциальными;
- токены и учетные данные не должны жестко вшиваться в клиентский код;
- секреты должны храниться в конфигурации окружения или безопасном серверном контуре;
- доступ к данным должен определяться правами источника Architeezy, а не дублирующей логикой в MVP;
- нельзя отправлять модельные данные во внешние SaaS-сервисы без явного решения на это.

Для первой версии не обязательно строить сложную security platform, но эти принципы должны влиять на архитектуру с самого начала.

### Нужно ли поддерживать работу с конфиденциальными архитектурными моделями без передачи данных во внешние сервисы

Да, это нужно считать **обязательным требованием MVP**.

Причина очень практическая:

- архитектурные модели часто содержат чувствительную информацию о системах, данных и зависимостях;
- если продукт с самого начала предполагает вынос данных во внешние сервисы, многие команды просто не смогут его использовать.

Итоговое требование:

> MVP должен работать локально или в контролируемом окружении без обязательной передачи архитектурных данных во внешние облачные сервисы.

### Какие требования к логированию и диагностике нужны для MVP

Для MVP нужен минимальный, но полезный диагностический слой:

- логирование статусов `load / index / analyze / export`;
- логирование ошибок интеграции;
- логирование предупреждений о неполных или неконсистентных данных;
- отображение понятных ошибок в UI;
- возможность понять, на каком этапе сломался сценарий.

Не нужен enterprise-observability stack, но нужна достаточная диагностика для разработки, отладки и доверия пользователей.

### Нужны ли требования к объяснимости метрик как отдельное нефункциональное качество

Да, **объяснимость — это отдельное нефункциональное качество** и одно из ключевых для ArchiLens OSS.

Для MVP это означает:

- каждая базовая метрика должна иметь понятное определение;
- пользователь должен видеть, почему элемент попал в impact result или quality issue;
- risk summary не должен быть black box;
- документация должна объяснять правила расчета хотя бы на уровне продукта.

Это почти так же важно, как производительность, потому что продукт продает не просто вычисление, а доверяемый анализ.

### Какой уровень доступности/Accessibility нужен для первой версии

Для MVP нужен **базовый уровень accessibility**, но без обещаний полного enterprise-grade покрытия.

Минимум:

- читаемая типографика и контраст;
- управление ключевыми действиями без точного попадания мышью в мелкие элементы;
- корректные подписи для основных интерактивных контролов;
- usable работа таблицы и основных форм.

Сложные графовые визуализации полностью доступными сделать трудно, поэтому здесь допустим `best effort`, но ключевые результаты не должны быть доступны только через визуальный граф.

### Насколько важна расширяемость API/плагинов уже в архитектуре MVP

Сами плагины в MVP не нужны, но **архитектурная готовность к расширению важна уже сейчас**.

Практический вывод:

- engine должен быть отделен от UI;
- расчет метрик должен быть модульным;
- слой интеграции с Architeezy не должен быть размазан по всему приложению;
- экспорт и quality rules должны иметь точку расширения, даже если она еще не оформлена как публичный API.

То есть расширяемость — это `NFR`, влияющий на структуру кода, а не обязательство делать plugin system прямо сейчас.

### Какие требования к сопровождаемости

Для MVP нужно зафиксировать реалистичный standard of quality:

- модульная структура кода с разделением `connector / graph engine / insight logic / UI`;
- тесты на graph normalization и core analysis logic;
- базовые интеграционные проверки ключевого сценария;
- документация запуска и архитектурного устройства;
- единый кодстайл и линтинг.

Что пока не обязательно:

- полная API-стабильность публичного extension API;
- экстремально высокий coverage ради метрики.

Хороший ориентир:

> Самые рискованные части логики должны быть покрыты тестами, а структура проекта должна позволять безопасно расширять продукт без распада архитектуры.

### Есть ли требования к лицензии, юридической чистоте зависимостей и open source governance

Да, эти требования нужно считать обязательными с самого начала.

Базовые решения:

- выбрать понятную OSS-лицензию с учетом желаемого режима использования и вкладов сообщества;
- избегать зависимостей с конфликтующей или сомнительной лицензией;
- вести список ключевых зависимостей и их лицензий;
- не включать в open source репозиторий данные или код, на которые нет права публикации;
- отделить OSS-ядро от любых потенциально чувствительных интеграционных артефактов.

Для первой версии достаточно легкого governance-пакета: `LICENSE`, `README`, правила вкладов, базовая политика по зависимостям.

## Итоговая нефункциональная рамка MVP

MVP ArchiLens OSS должен уверенно работать на моделях малого и среднего размера, давать результат impact analysis за секунды, быть desktop-first веб-приложением, устойчивым к неполным данным и временным сбоям API, не выводить архитектурные данные во внешние сервисы без необходимости и обеспечивать объяснимость метрик как ключевое качество продукта. Архитектура должна сразу закладывать модульность, тестируемость и будущую расширяемость, даже если сами расширения появятся позже.
