# 05_Functional_Requirements: ответы и принятые решения

## Принцип формулирования требований

На этом этапе нам уже недостаточно модулей и красивых названий экранов. Нужны атомарные функциональные требования, которые можно проверить по факту работы системы. Для первой версии удобнее всего использовать приоритизацию `P0 / P1 / P2`, потому что она жестко отделяет MVP-обязательства от полезных, но отложенных функций.

Правило простое:

- `P0` — без этого MVP не выполняет главную работу;
- `P1` — усиливает продукт, но MVP может быть принят и без этого;
- `P2` — post-MVP или явный резерв.

## Ответы на вопросы

### Какой формат приоритизации требований нужно использовать

Оптимально использовать **`P0 / P1 / P2`**.

Почему не `Must / Should / Could`:

- по смыслу они похожи;
- но `P0 / P1 / P2` лучше ложится на backlog, приемку и инженерную дискуссию;
- `P0` психологически жестче фиксирует обязательный контур MVP.

### Какие функциональные требования являются обязательными для главного MVP-сценария

Ключевые `P0` требования для MVP:

- система должна подключаться к одному источнику Architeezy или открывать demo dataset;
- система должна загружать одну выбранную модель;
- система должна автоматически строить единый граф модели;
- система должна индексировать элементы для поиска и навигации;
- система должна позволять найти элемент по имени;
- система должна позволять выбрать элемент из таблицы или графа;
- система должна запускать impact analysis на глубину 1, 2 или 3 шага;
- система должна показывать затронутые элементы, слои и диаграммы;
- система должна показывать базовые метрики выбранного элемента;
- система должна показывать orphan и low-coverage элементы;
- система должна поддерживать экспорт результата impact analysis в `CSV` и `GraphML`.

Если любой из этих пунктов не реализован, главный сценарий MVP считается незавершенным.

### Должна ли система уметь подключаться к нескольким экземплярам Architeezy или только к одному

Для MVP достаточно **одного настроенного экземпляра Architeezy**.

Поддержка нескольких экземпляров — это `P1`, потому что:

- не влияет на подтверждение гипотезы;
- увеличивает сложность настройки и тестирования;
- создает лишние вопросы по контексту подключения, безопасности и UX выбора источника.

### Должна ли система сохранять настройки подключения между сессиями

В MVP это может быть решено минималистично.

`P0`:

- система должна уметь работать с уже заданной конфигурацией подключения.

`P1`:

- система может сохранять пользовательские настройки подключения между сессиями.

Итоговое решение:

> Для первой версии не требуется полноценный UI управления подключениями. Достаточно конфигурации на уровне окружения или простой локальной настройки.

### Какие сущности из Architeezy MVP обязан загружать

`P0` набор сущностей:

- модели;
- элементы;
- отношения;
- диаграммы или представления;
- ссылки между элементами и диаграммами;
- типы элементов;
- базовые метаданные, необходимые для идентификации и отображения.

`P1`:

- дополнительные свойства элементов;
- теги;
- пользовательские атрибуты;
- расширенные metadata blocks.

Для MVP важно загружать не “все что можно”, а именно тот минимум, который нужен для impact, coverage и explainability.

### Должна ли система индексировать модель автоматически после загрузки или только по явному действию пользователя

Да, система должна **автоматически индексировать модель после загрузки**. Это `P0`.

Почему:

- ручной шаг индексирования ломает основной поток;
- пользователь не должен разбираться во внутренней подготовке графа;
- для MVP важен быстрый путь от выбора модели к анализу.

Допустим промежуточный статус: `loading -> indexing -> ready`.

### Какие функции поиска обязательны

`P0` поиск:

- по имени элемента;
- по типу элемента;
- по слою через фильтр;
- по ID элемента, если ID известен и доступен в данных.

`P1`:

- по диаграмме;
- по тегам;
- по пользовательским свойствам.

Приоритет должен быть в пользу простоты и скорости. Поиск по имени — главный рабочий путь.

### Какие фильтры обязательны для Global Graph и таблицы

`P0` фильтры:

- по слою;
- по типу элемента;
- по принадлежности к результату impact analysis;
- по проблемности в coverage/hygiene режиме.

`P1`:

- по диаграмме;
- по наличию/отсутствию диаграммного покрытия;
- по направлению связей в таблице;
- по произвольным пользовательским атрибутам.

### Какие базовые метрики система обязана считать в MVP, и в какой момент они рассчитываются

`P0` метрики:

- `degree`;
- `in-degree`;
- `out-degree`;
- `diagrams count`;
- `orphan flag`;
- `low-coverage flag`;
- `impact reach` в рамках выбранной глубины.

Когда считать:

- структурные метрики и coverage-индикаторы — сразу после загрузки и построения графа;
- `impact reach` — по запросу при запуске impact analysis.

Это позволяет не перегружать систему ненужными расчетами, но при этом быстро отдавать базовый обзор модели.

### Какие именно действия должен поддерживать Impact Analyzer

`P0` действия Impact Analyzer:

- выбор исходного элемента;
- выбор глубины анализа 1/2/3;
- расчет затронутых сущностей;
- показ затронутых слоев;
- показ затронутых диаграмм;
- переключение между табличным и графовым представлением результата;
- переход к связанному элементу;
- экспорт результата.

`P1`:

- переключение направления обхода `outgoing / incoming / both`, если это не усложняет MVP чрезмерно;
- сохранение набора фильтров внутри сессии.

Если нужно выбирать, что урезать первым, то менее критично направление обхода, но не сам экспорт и не просмотр диаграмм.

### Какие действия обязательны для coverage-анализа

`P0`:

- показать список orphan elements;
- показать список low-coverage elements;
- дать перейти к элементу из списка;
- дать экспортировать список проблемных элементов в `CSV`.

`P1`:

- агрегированные heatmaps;
- настраиваемые quality thresholds;
- сравнительные представления по слоям.

### Нужна ли пользователю возможность сравнить несколько элементов между собой уже в MVP

Нет, это не нужно в MVP. Это `P2`.

Почему:

- не относится к главной гипотезе;
- усложняет UX и визуализацию;
- лучше сначала хорошо реализовать анализ одного элемента, чем посредственно сравнение нескольких.

### Должен ли экспорт поддерживать только текущий выделенный подграф или также таблицу, снимок анализа и изображение

`P0` экспорт:

- текущий выделенный подграф в `GraphML`;
- табличный результат в `CSV`.

`P1`:

- снимок анализа в `JSON`;
- экспорт изображений `PNG/SVG`.

Итог:

> MVP обязан экспортировать подграф и таблицу. Все остальные форматы откладываются.

### Нужно ли в MVP поддерживать объяснение того, почему элемент помечен как критичный или проблемный

Да, в MVP нужна **объяснимость**, но в ограниченном объеме. Это `P0`.

Что именно должно быть объяснимо:

- почему элемент попал в impact result;
- почему он считается orphan или low-coverage;
- на каких базовых признаках строится краткий risk summary.

Не нужно в MVP строить сложный explainability framework. Достаточно прозрачного текстового или структурного объяснения рядом с результатом.

### Что система должна делать, если модель неполная, содержит сиротские ссылки, циклы, дубли или неконсистентные данные

Система должна быть **устойчивой к неидеальным данным**. Это `P0` требование.

Поведение должно быть таким:

- пытаться построить граф из доступных корректных данных;
- логировать и показывать предупреждения о проблемных участках;
- не падать из-за частичной неконсистентности;
- помечать неполные или подозрительные результаты, если это влияет на точность анализа.

То есть принцип такой: `best-effort analysis with explicit warnings`.

### Какие функции администрирования/настройки должны существовать в MVP, если вообще должны

Минимальный `P0` набор настройки:

- указание или чтение конфигурации подключения;
- выбор модели;
- базовый статус источника и состояния загрузки.

`P1` или позже:

- UI управления несколькими подключениями;
- роли и права пользователей;
- продвинутые настройки quality rules;
- управление сохраненными представлениями.

### Какие функции из идеи нужно явно классифицировать как `P1/P2`

`P1`:

- улучшенная risk card;
- поиск по расширенным атрибутам;
- фильтры по диаграмме;
- JSON snapshot export;
- direction toggles, если не входят в самый компактный MVP;
- улучшенные визуальные режимы подграфа.

`P2`:

- Criticality Radar как отдельный режим;
- Cross-Layer X-Ray;
- Compare States;
- Saved Views;
- Shareable Links;
- Plugin API;
- multi-model analysis;
- alerts;
- community extensions.

## Итоговая функциональная рамка MVP

Функциональное ядро ArchiLens OSS в MVP — это загрузка одной модели из Architeezy, автоматическое построение и индексирование графа, поиск элемента, impact analysis на глубину 1-3 шага, базовые метрики и coverage-индикаторы, а также экспорт подграфа и таблицы результата. Все функции, не усиливающие этот поток напрямую, должны быть классифицированы как `P1` или `P2`, чтобы не превратиться в скрытые обязательства релиза.
