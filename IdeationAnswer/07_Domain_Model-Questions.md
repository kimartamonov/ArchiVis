# 07_Domain_Model: ответы и принятые решения

## Принцип доменного моделирования

Сейчас важно не повторить структуру Architeezy один в один, а построить **каноническую аналитическую модель ArchiLens**. Она должна быть достаточно близкой к source data, чтобы не потерять смысл, и достаточно самостоятельной, чтобы поддерживать analysis use cases без хаоса в коде.

Ключевое разделение для MVP такое:

- есть `source-of-truth` сущности, пришедшие из Architeezy;
- есть `derived` сущности аналитического слоя;
- есть `user-facing artifacts`, которые продукт создает по итогам анализа.

## Ответы на вопросы

### Какие доменные сущности являются основными в MVP

Основные сущности MVP:

- `Model`;
- `Element`;
- `Relationship`;
- `View` или `Diagram`;
- `ElementViewRef` или `ElementPresentation`;
- `Layer`;
- `MetricSet`;
- `ImpactAnalysisRequest`;
- `ImpactAnalysisResult`;
- `Subgraph`;
- `ExportArtifact`;
- `DataIssue`.

Почему именно так:

- `Model`, `Element`, `Relationship`, `View` — базовый source layer;
- `ElementViewRef`, `MetricSet`, `ImpactAnalysisResult`, `Subgraph` — derived layer;
- `ExportArtifact` — пользовательский артефакт;
- `DataIssue` нужен, потому что неполные и битые данные для этого продукта являются нормальной реальностью, а не исключением.

### Что является единицей анализа

Главная единица анализа в MVP — **отдельный элемент ArchiMate внутри выбранной модели**.

Но важно понимать структуру результата:

- вход в анализ: `Element`;
- выход анализа: `Subgraph + ImpactAnalysisResult`.

То есть продукт начинает с одного элемента, но анализирует не его в вакууме, а его положение внутри графа.

### Какие обязательные поля должны быть у сущности `Element` в канонической модели

Минимальный обязательный набор полей `Element`:

- `sourceId` — идентификатор в Architeezy;
- `modelId` — принадлежность модели;
- `name`;
- `type` — ArchiMate element type;
- `layer`;
- `documentation` или краткое описание, если доступно;
- `viewRefs` — ссылки на представления элемента в диаграммах;
- `properties` — только как необязательный контейнер расширения;
- `statusFlags` — служебные derived-флаги вроде `isOrphan`, `isLowCoverage`.

Важно не перегружать `Element` всеми возможными данными источника. Каноническая модель должна включать только то, что реально участвует в поиске, анализе, объяснении и навигации.

### Как идентифицируются сущности

Для MVP лучше использовать двухуровневую идентичность:

- основная identity: `sourceId` из Architeezy + `modelId`;
- внутренняя техническая identity внутри аналитического слоя: нормализованный `nodeId` или `edgeId`.

Почему не только sourceId:

- один и тот же технический формат ID может быть неуниверсален для всех будущих источников;
- нормализованный внутренний идентификатор упрощает работу graph engine.

Итог:

> Канонический ключ сущности в MVP — составной: `modelId + sourceId`, а для внутренних операций допускается собственный stable internal ID.

### Как представляются типы ArchiMate и слои в доменной модели

Нужно разделить два уровня:

- `type` — конкретный тип сущности ArchiMate, например `ApplicationComponent`, `BusinessProcess`, `DataObject`;
- `layer` — укрупненная принадлежность к слою, например `Business`, `Application`, `Technology`, `Data`, `Motivation`.

Это важно, потому что:

- тип нужен для точной навигации и объяснения;
- слой нужен для фильтров, агрегатов и impact summary.

Не стоит пытаться свести тип и слой в одно поле.

### Нужна ли отдельная сущность для диаграммного представления элемента, отличная от самого элемента

Да, для MVP нужна отдельная derived-сущность или структурная связь вроде **`ElementViewRef`**.

Почему это обязательно:

- один и тот же элемент может встречаться в нескольких диаграммах;
- сам элемент и его представление на диаграмме — не одно и то же;
- coverage-анализ невозможно корректно считать без этого разделения.

`ElementViewRef` может быть легковесной сущностью с полями:

- `elementId`;
- `viewId`;
- `presentationId`, если источник это дает;
- опциональные presentation metadata.

### Как моделируются производные метрики

Для MVP метрики лучше моделировать как отдельную сущность **`MetricSet`**, привязанную к элементу и контексту расчета.

Почему это лучше, чем просто вычисляемые поля внутри `Element`:

- часть метрик статична для модели, а часть зависит от запроса;
- это отделяет source data от derived analytics;
- упрощает тестирование и потенциальное расширение.

Практический подход:

- `ElementMetricSet` для предрассчитанных базовых метрик;
- impact-specific поля хранятся в `ImpactAnalysisResult`.

### Нужна ли сущность `AnalysisSession` или `AnalysisRequest`

Для MVP достаточно сущности **`ImpactAnalysisRequest`**, а полноценная `AnalysisSession` не обязательна.

Почему:

- результат реально зависит от параметров запроса: исходный элемент, глубина, возможно направление и фильтры;
- но постоянные пользовательские сессии анализа и сохранение истории пока не входят в MVP.

Минимальные поля `ImpactAnalysisRequest`:

- `modelId`;
- `elementId`;
- `depth`;
- `direction`, если поддерживается;
- `timestamp`.

### Как хранить identity элемента, если один и тот же объект встречается в нескольких view

Идентичность элемента должна оставаться **единой на уровне модели**, независимо от числа view.

Правильная схема:

- один `Element`;
- много `ElementViewRef`;
- много `View`.

Это критично, потому что иначе один и тот же архитектурный объект распадется на несколько ложных сущностей, а impact analysis и coverage потеряют смысл.

### Как моделировать битые или неполные ссылки из источника данных

Для MVP нужна сущность или как минимум тип записи **`DataIssue`**.

Она должна фиксировать:

- тип проблемы: missing target, broken view ref, duplicate, invalid relation и т.д.;
- уровень: model / element / relationship / view;
- связанный source ID;
- severity: warning или error;
- сообщение для диагностики.

Почему это важно:

- неидеальные данные — часть предметной области;
- без явной модели ошибок система будет либо падать, либо скрывать проблему.

### Какие пользовательские артефакты должны войти в доменную модель уже в MVP

В MVP достаточно только одного полноценного пользовательского артефакта: **`ExportArtifact`**.

В него входят:

- тип экспорта: `CSV` или `GraphML`;
- контекст создания: impact result или coverage list;
- timestamp;
- ссылка на источник данных результата.

Не нужны в MVP:

- сохраненные фильтры;
- сохраненные анализы;
- комментарии;
- shareable views.

### Где проходит граница между source-of-truth сущностями Architeezy и derived сущностями ArchiLens

Граница должна быть очень четкой.

`Source-of-truth`:

- `Model`;
- `Element`;
- `Relationship`;
- `View`;
- исходные metadata fields.

`Derived`:

- `Layer`, если он выводится из типа;
- `ElementViewRef`, если нормализуется из исходных представлений;
- `MetricSet`;
- `ImpactAnalysisRequest`;
- `ImpactAnalysisResult`;
- `Subgraph`;
- `DataIssue`;
- `ExportArtifact`.

Ключевой принцип:

> Architeezy сообщает, что существует в репозитории. ArchiLens добавляет то, что нужно для анализа, объяснения и экспорта.

### Какие инварианты предметной модели обязательны

Обязательные инварианты MVP:

- `Element` уникален в рамках `modelId + sourceId`;
- `Relationship` ссылается на допустимые source и target элементы либо помечается как `DataIssue`;
- `View` принадлежит одной модели;
- `ElementViewRef` связывает существующий `Element` и существующий `View` либо фиксируется как проблема данных;
- каждый `Element` имеет определенный `type`;
- каждый `Element` имеет определенный `layer` явно или через детерминированное отображение;
- `ImpactAnalysisResult` всегда связан с конкретным `ImpactAnalysisRequest`.

Эти инварианты нужны и для корректности кода, и для объяснимости результатов.

### Нужны ли версии/снимки модели как доменная сущность уже в MVP

Нет, полноценные версии или snapshots модели как отдельная доменная сущность в MVP не нужны.

Почему:

- сравнение состояний не входит в первую версию;
- хранение снапшотов быстро утяжеляет модель и архитектуру;
- для MVP достаточно работать с текущим состоянием выбранной модели.

Единственное допустимое упрощение:

- хранить технический `loadedAt` или `sourceRevision`, если источник это предоставляет.

### Какие сущности точно не нужны в MVP

Чтобы не переусложнить модель, в MVP точно не нужны:

- `AnalysisSession` с историей пользовательских действий;
- `SavedView`;
- `Comment`;
- `Annotation`;
- `ShareLink`;
- `ModelVersion` как полноценный агрегат;
- `Plugin`;
- `RuleDefinitionEditor`;
- `ComparisonResult` между версиями;
- `CollaborationThread`.

## Итоговая доменная модель MVP

В основе MVP ArchiLens OSS лежит выбранная `Model`, содержащая канонические `Element`, `Relationship` и `View`. Один элемент остается одной сущностью независимо от числа диаграммных представлений, а связи элемента с диаграммами выражаются через `ElementViewRef`. Аналитический слой строится поверх source data и включает `MetricSet`, `ImpactAnalysisRequest`, `ImpactAnalysisResult`, `Subgraph`, `DataIssue` и `ExportArtifact`. Эта модель достаточно богата для impact analysis и coverage, но еще не перегружена сущностями сохранения, коллаборации и версионирования.
