# 04_User_Scenarios_and_Use_Cases: ответы и принятые решения

## Принцип сценарного проектирования

Для MVP нельзя одинаково подробно описывать все use cases. Нужен один канонический end-to-end поток, вокруг которого строятся UX, архитектура, приемка и демонстрация. Все остальные сценарии должны либо усиливать этот поток, либо явно быть отложены.

Для ArchiLens OSS таким каноническим сценарием должен стать не обзор модели вообще, а **impact analysis конкретного элемента на реальной модели**.

## Ответы на вопросы

### Какой один end-to-end сценарий должен стать главным сценарием MVP и основой demo script

Главный сценарий MVP:

> Архитектор выбирает элемент модели, запускает impact analysis, получает список затронутых сущностей, слоев и диаграмм, оценивает риск и выгружает результат для обсуждения изменений.

Почему именно он:

- он лучше всего подтверждает главную гипотезу MVP;
- он понятен и архитектору, и наблюдателю со стороны;
- он тянет за собой почти все обязательные части продукта: загрузку модели, поиск, граф, метрики, explainability, export.

### С какого экрана и с какого триггера пользователь начинает каждый ключевой сценарий

Нужно зафиксировать единый стартовый принцип:

- пользователь начинает с `Model Explorer` или стартового экрана выбора модели;
- после выбора модели попадает в контекст модели;
- дальше входит в сценарий либо через поиск, либо через таблицу элементов.

Для ключевых сценариев это выглядит так:

- `Impact Analysis`: старт из поиска элемента, таблицы или клика по узлу в графе;
- `Model Coverage & Hygiene`: старт из раздела `Coverage & Quality` после загрузки модели;
- `Evidence-based narrative`: старт из завершенного impact analysis, а не как отдельная точка входа.

### Какие предусловия обязательны для начала сценария

Обязательные предусловия для основного сценария:

- настроено подключение к Architeezy или загружен demo dataset;
- пользователь выбрал одну модель;
- модель успешно загружена;
- единый граф уже построен и индексирован;
- элемент либо известен заранее, либо может быть найден через UI.

Если хотя бы одно из этих предусловий не выполнено, нельзя делать вид, что сценарий доступен. Система должна явно сообщать, на каком шаге остановка.

### Как пользователь находит нужный элемент

Для MVP оптимальный приоритет поиска такой:

1. Поиск по имени элемента.
2. Выбор из таблицы элементов с фильтрацией.
3. Клик по узлу на графе.

Не стоит делать граф первичной точкой входа. Для реальной работы архитектору чаще нужен быстрый поиск известной сущности, а не исследование визуального полотна.

`Deeplink` можно отложить. Он полезен, но не обязателен для первой версии.

### Какие шаги пользователь проходит в сценарии `Impact Analysis`

Канонический поток `Impact Analysis` должен выглядеть так:

1. Пользователь выбирает модель.
2. Система строит и подготавливает единый граф.
3. Пользователь находит элемент через поиск, таблицу или граф.
4. Открывает карточку элемента.
5. Выбирает глубину анализа: 1, 2 или 3 шага.
6. Запускает impact analysis.
7. Система вычисляет затронутые узлы и связи.
8. Пользователь получает структурированный результат.
9. При необходимости фильтрует или просматривает результат в графовом и табличном виде.
10. Экспортирует результат наружу.

Это и есть главная сквозная история MVP.

### Какие именно выходы должны появляться по завершении `Impact Analysis`

По завершении `Impact Analysis` пользователь должен получить минимум следующие outputs:

- список `affected elements`;
- агрегированный список `affected layers`;
- список `affected diagrams`;
- базовые показатели связанности для исходного элемента;
- краткую `risk card`;
- визуальный подграф результата;
- возможность экспорта в `CSV` и `GraphML`.

Важно: risk card в MVP не должна быть “умным black box”. Это должен быть короткий, объяснимый summary на основе прозрачных признаков.

### Какой шаг за шагом сценарий у `Model Coverage & Hygiene`

Сценарий `Model Coverage & Hygiene` должен быть вторым по важности и максимально простым:

1. Пользователь выбирает модель.
2. Система рассчитывает coverage-индикаторы после построения графа.
3. Пользователь открывает экран `Coverage & Quality`.
4. Видит summary по orphan elements и low-coverage elements.
5. Может провалиться в список проблемных сущностей.
6. Может выбрать проблемный элемент и перейти к его карточке или impact analysis.
7. Может выгрузить список проблемных элементов в `CSV`.

То есть это не самостоятельный огромный модуль, а diagnostic layer поверх уже загруженной модели.

### Нужен ли отдельный пошаговый сценарий для `Criticality Radar` уже в MVP или это post-MVP use case

Это **post-MVP use case**.

В первой версии не нужно делать отдельный продуктовый поток `Criticality Radar`, потому что:

- для него нужны более зрелые метрики;
- он повышает риск “магических” выводов;
- он не нужен для проверки главной гипотезы.

Максимум, что допустимо в MVP, — показывать базовые метрики связанности в таблице без выделения отдельного режима criticality.

### Как выглядит сценарий подготовки evidence-based narrative для стейкхолдеров

В MVP это должен быть облегченный, но практичный сценарий:

1. Архитектор завершает impact analysis.
2. Отбирает релевантный подграф или табличный результат.
3. Получает:
   - список затронутых элементов;
   - перечень затронутых диаграмм;
   - краткий риск-контекст;
   - экспортируемый табличный артефакт.
4. Использует это как основу для обсуждения на review или change board.

Важно не превращать MVP в storytelling-платформу. На старте продукт должен давать **сырье для аргументации**, а не сложный конструктор презентаций.

### Какие сценарии должны поддерживать экспорт, а какие только просмотр внутри UI

Экспорт обязателен для:

- `Impact Analysis`;
- `Model Coverage & Hygiene` в табличной части.

Только просмотр внутри UI допустим для:

- `Global Graph` как обзорного режима;
- карточки элемента;
- навигации по модели.

Иначе говоря, экспорт нужен там, где результат выходит за пределы самого продукта и идет в коммуникацию или внешнюю обработку.

### Какие пользовательские ошибки и отклонения нужно заранее описать

Для MVP нужно явно описать и обработать следующие отклонения:

- элемент не найден;
- модель не выбрана;
- данные модели загружены не полностью;
- Architeezy API не отвечает;
- граф не удалось построить;
- модель слишком большая для полного отображения;
- для части элементов отсутствуют диаграммные ссылки;
- экспорт не может быть сформирован.

Во всех этих случаях продукт должен не просто падать или показывать generic error, а объяснять, что именно недоступно и что пользователь может сделать дальше.

### Что считается успешным завершением каждого сценария в проверяемом виде

`Impact Analysis` считается успешно завершенным, если:

- пользователь выбрал элемент;
- получил результат на выбранной глубине;
- увидел затронутые сущности, слои и диаграммы;
- смог экспортировать результат.

`Model Coverage & Hygiene` считается успешно завершенным, если:

- пользователь увидел список orphan и low-coverage элементов;
- смог перейти к конкретной сущности;
- смог выгрузить проблемный список наружу.

`Evidence-based narrative` считается успешно поддержанным, если:

- архитектор может вынести из системы понятный артефакт для обсуждения, не собирая данные вручную из множества диаграмм.

### Какие сценарии являются основными, какие второстепенными, а какие только stretch

Основной сценарий:

- `Impact Analysis`.

Второстепенные, но входящие в MVP:

- `Global Graph` как навигационный режим;
- `Таблица элементов`;
- `Model Coverage & Hygiene`;
- `Экспорт результатов`.

Stretch, но не обязательные:

- более умная risk card;
- переходы между несколькими режимами визуализации impact-результата;
- улучшенный evidence pack.

Post-MVP:

- `Criticality Radar` как отдельный режим;
- `Cross-Layer X-Ray` как самостоятельный сценарий;
- `Compare States`;
- `Saved Views`;
- `Shareable Links`.

### Какие сценарии должны поддерживать работу на реальной модели, а какие можно показывать только на demo dataset

На реальной модели обязаны работать:

- загрузка модели;
- поиск элемента;
- impact analysis;
- coverage/hygiene;
- экспорт.

На demo dataset можно дополнительно показывать:

- идеальный демонстрационный граф;
- более “чистый” walkthrough для презентации;
- примеры сценариев, где данные в реальной модели пока недостаточно качественные.

Но нельзя допустить, чтобы ключевая ценность продукта существовала только на demo dataset.

### Какие сценарии нужно явно исключить из MVP

Из MVP нужно явно исключить следующие сценарии:

- редактирование модели;
- сравнение двух состояний модели;
- совместную работу над аналитическим view;
- генерацию управленческих отчетов “под ключ”;
- deep executive storytelling UI;
- multi-model analysis;
- оповещения и подписки на деградацию модели;
- пользовательские плагины и конструктор правил.

## Итоговый сценарный каркас MVP

MVP ArchiLens OSS строится вокруг одного канонического потока: выбрать модель, найти элемент, запустить impact analysis, получить объяснимый результат и выгрузить его для обсуждения изменений. `Coverage & Hygiene` и табличный режим являются поддерживающими сценариями, а не самостоятельными центрами продукта. Все сценарии, не усиливающие эту основную историю, выносятся за пределы первой версии.
