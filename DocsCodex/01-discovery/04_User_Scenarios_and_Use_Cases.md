# 04 User Scenarios and Use Cases

## Основной принцип

В MVP ArchiLens OSS есть один главный сквозной сценарий, вокруг которого должны строиться UX, требования, приемка и демонстрация. Этот сценарий — impact analysis выбранного элемента на реальной модели.

## Главный end-to-end сценарий MVP

### Название

Impact Analysis for Selected Element.

### Цель пользователя

Понять, что затронет изменение конкретного элемента ArchiMate-модели, и получить артефакт для дальнейшего обсуждения.

### Актор

Solution Architect / Enterprise Architect.

### Предусловия

- настроено подключение к Architeezy или открыт demo dataset;
- выбрана одна модель;
- модель загружена;
- канонический граф построен и индексирован.

### Основной поток

1. Пользователь открывает Model Explorer.
2. Выбирает модель.
3. Система загружает данные и строит единый граф.
4. Пользователь находит элемент через поиск, таблицу или граф.
5. Открывает карточку элемента.
6. Выбирает глубину анализа: 1, 2 или 3.
7. Запускает impact analysis.
8. Система вычисляет затронутые элементы и связи.
9. Пользователь получает:
   - affected elements;
   - affected layers;
   - affected diagrams;
   - базовые метрики;
   - краткий risk summary.
10. Пользователь при необходимости переключается между табличным и графовым представлением.
11. Экспортирует результат в CSV или GraphML.

### Ожидаемый результат

Пользователь получает объяснимый и пригодный для обсуждения результат анализа влияния.

### Критерий успешности

Сценарий считается успешно пройденным, если пользователь дошел от выбора модели до экспортируемого результата impact analysis без ручного обхода диаграмм вне системы.

## Поддерживающий сценарий 1: Model Coverage and Hygiene

### Цель

Понять, где модель деградирует и какие элементы представлены недостаточно.

### Поток

1. Пользователь выбирает модель.
2. Система рассчитывает базовые quality indicators после построения графа.
3. Пользователь открывает Coverage and Quality.
4. Видит:
   - orphan elements;
   - low-coverage elements.
5. Переходит к проблемному элементу.
6. При необходимости запускает для него impact analysis.
7. Может экспортировать проблемный список в CSV.

### Ценность

Этот сценарий усиливает governance use case и повышает регулярную ценность продукта между change-сессиями.

## Поддерживающий сценарий 2: Evidence-Based Narrative Preparation

### Цель

Подготовить материал для review, change board или объяснения стейкхолдерам.

### Поток

1. Пользователь завершает impact analysis.
2. Отбирает релевантный подграф или список затронутых элементов.
3. Получает:
   - табличный экспорт;
   - графовый экспорт;
   - краткий риск-контекст;
   - список связанных диаграмм.
4. Использует этот набор как evidence pack.

### Ценность

Продукт помогает не только исследовать архитектуру, но и доносить выводы.

## Точки входа в сценарии

Основные входы:

- выбор модели через стартовый экран;
- поиск элемента по имени;
- выбор элемента из таблицы;
- клик по узлу в графе.

Primary entry point для MVP — поиск и таблица, а не граф.

## Выходы сценариев

### Для Impact Analysis

Обязательные outputs:

- список affected elements;
- список affected layers;
- список affected diagrams;
- подграф результата;
- базовые метрики seed-элемента;
- export в CSV и GraphML.

### Для Coverage and Hygiene

Обязательные outputs:

- список orphan elements;
- список low-coverage elements;
- переход к конкретному элементу;
- экспорт в CSV.

## Ошибки и отклонения, которые должны поддерживаться

Система должна явно обрабатывать:

- элемент не найден;
- модель не выбрана;
- модель загружена частично;
- API недоступен;
- граф не построился;
- модель слишком тяжелая для полного отображения;
- данные содержат broken references;
- экспорт не может быть сформирован.

Во всех случаях пользователь должен получить понятное объяснение, на каком шаге возникла проблема.

## Сценарии, исключенные из MVP

В MVP не входят как полноценные use cases:

- compare two states of model;
- multi-model analysis;
- collaborative views;
- saved analysis sessions;
- shareable links;
- advanced Criticality Radar как самостоятельный режим;
- Cross-Layer X-Ray как отдельный end-to-end поток.

## Вывод

Главный сценарий ArchiLens OSS — это быстрый, объяснимый impact analysis выбранного элемента. Все остальные сценарии в MVP либо поддерживают этот поток, либо усиливают его ценность в governance и коммуникации. Именно от этого документа дальше трассируются функциональные требования, acceptance criteria и capability slices.
