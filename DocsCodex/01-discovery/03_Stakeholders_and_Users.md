# 03 Stakeholders and Users

## Основной фокус документа

ArchiLens OSS нельзя проектировать одновременно для всех ролей, связанных с архитектурой. Для MVP нужен один primary user, один critical secondary stakeholder и понятное понимание того, как остальные роли потребляют результат.

## Основные заинтересованные стороны

### Внутренние для проекта

- Product Owner ArchiLens OSS;
- Tech Lead / Solution Architect проекта;
- потенциальные contributors open source проекта.

### Внешние для ценности продукта

- Solution Architect;
- Enterprise Architect;
- Repository Owner / EA Lead;
- Transformation Manager;
- руководители и стейкхолдеры, которым нужен понятный impact overview;
- команда Architeezy как экосистемный стейкхолдер.

## Primary user MVP

### Роль

Primary user — Solution Architect или Enterprise Architect, который регулярно оценивает последствия изменений.

### Рабочий контекст

Этот пользователь:

- работает с уже существующей ArchiMate-моделью;
- получает change requests, review-запросы или governance-задачи;
- знает предметную область, но не может держать в голове всю модель целиком;
- испытывает давление по времени и должен быстро собрать доказуемый анализ.

### Его главные задачи

- понять, что затронет изменение выбранного элемента;
- определить, какие домены и команды надо вовлечь;
- оценить риск побочных эффектов;
- подготовить материал для review или change board.

### Его боли

- последствия изменений трудно видеть сквозным образом;
- знания о зависимостях часто неявны и распределены между экспертами;
- модель есть, но пользоваться ею для быстрых решений неудобно;
- диаграммы не дают целостного ответа на вопрос “что сломается”.

## Critical secondary stakeholder

### Роль

Repository Owner / EA Lead.

### Почему роль критична

Этот участник отвечает не только за ценность отдельного анализа, но и за то, станет ли инструмент частью регулярной практики команды.

### Его боли

- деградация модели незаметна до тех пор, пока не становится критичной;
- трудно системно видеть сиротские элементы и слабое покрытие;
- репозиторий рискует стать архивом вместо рабочего инструмента.

## Контекст целевой организации

Целевой контекст MVP:

- средняя или крупная организация;
- есть архитектурный офис, enterprise architecture practice или transformation team;
- ведется ArchiMate-репозиторий в Architeezy;
- у команды есть регулярные change, review и governance-сценарии.

Менее приоритетный контекст:

- маленькие компании без зрелого репозитория;
- purely consulting artifacts без регулярного использования модели;
- среды, где архитектурная модель почти не влияет на решения.

## Зрелость репозитория

Целевой репозиторий для MVP:

- уже существует и используется несколькими людьми;
- содержит несколько слоев ArchiMate;
- имеет диаграммы и view-level представления;
- достаточно крупный, чтобы ручной анализ был дорогим;
- неидеален по качеству: есть coverage gaps, orphan cases, неоднородная детализация.

## Частота использования продукта

Продукт предполагает регулярное, но event-driven использование:

- по change request;
- перед architecture review;
- перед governance session;
- перед подготовкой объяснения для стейкхолдеров.

Это означает, что UX должен быть оптимизирован под быстрый вход, быстрый анализ и быстрый экспорт, а не под длительную непрерывную работу.

## Уровень экспертизы пользователя

По умолчанию MVP проектируется для пользователя со средним или высоким уровнем архитектурной и ArchiMate-компетенции.

Продукт не должен требовать обучения базовой нотации, но должен помогать в сквозном анализе сложной модели.

## Как вторичные роли потребляют результат

### Transformation Manager

Потребляет не весь analysis UI, а evidence-based результат:

- список затронутых элементов;
- краткий риск-контекст;
- выгруженный подграф или таблицу.

### Руководитель или sponsor

Потребляет результат через:

- объяснимый summary;
- табличный вывод;
- сопровождение со стороны архитектора.

### Команда Architeezy

Видит продукт как:

- proof of value их API;
- экосистемный слой анализа;
- не конкурента core repository/editing продукту.

## Ограничения пользователей

Пользовательские ограничения, важные для MVP:

- дефицит времени;
- неполное знание всей модели;
- недоверие к black-box score;
- необходимость выносить результат во внешние коммуникации;
- неидеальное качество исходной модели.

## Кто принимает результат MVP

С практической стороны MVP принимают:

- архитектор-лид;
- Repository Owner / EA Lead;
- вторая линия проверки — архитектурный эксперт, который может оценить правдоподобность результатов анализа.

## Допустимые продуктовые компромиссы

В первой версии допустимо:

- меньше гибкости ради скорости;
- меньше визуальной зрелищности ради полезности;
- меньше охватываемых ролей ради четкого ICP;
- меньше сложных метрик ради доверия к результату.

Недопустимо только одно: потерять достоверность и объяснимость impact analysis.

## Вывод

ArchiLens OSS строится в первую очередь для архитектора, который должен быстро и уверенно оценить последствия изменений. Все остальные роли либо помогают принять и внедрить продукт, либо потребляют его результаты в более простом виде. Именно это должно определять приоритеты UX, требований и delivery.
