# 09 Data and Integrations

## Цель документа

Зафиксировать, как данные приходят из Architeezy, как они нормализуются, валидируются, живут внутри ArchiLens OSS и во что экспортируются наружу.

## Источник истины

### Для модели

Source of truth для архитектурной модели — Architeezy.

### Для аналитических результатов

Source of truth для impact analysis и quality indicators внутри одной загрузки — пересчитанный канонический граф ArchiLens, построенный из данных Architeezy.

Это означает, что derived results не заменяют источник и не должны восприниматься как отдельная master data система.

## Нужные ресурсы Architeezy для MVP

Для первой версии нужны данные по:

- списку моделей;
- выбранной модели;
- элементам модели;
- отношениям модели;
- диаграммам или представлениям;
- связям элементов с diagram/view representations;
- базовым metadata fields.

## Подход к интеграции

Если существует стабильный OpenAPI/Swagger, generated client является предпочтительным стартовым вариантом. Но connector layer не должен зависеть только от generated code.

Обязателен собственный нормализующий контракт между connector и graph engine.

## Аутентификация и авторизация

Для MVP предполагается server-side configured authentication:

- credentials и токены управляются на стороне BFF/connector;
- UI не является хранилищем секретов;
- доступ к данным по возможности наследуется от правил доступа Architeezy.

## Ограничения интеграции, которые учитываются заранее

Нужно проектировать connector с учетом возможных проблем:

- pagination;
- rate limits;
- нестабильности схемы API;
- неполных связей между сущностями;
- больших ответов на уровне модели;
- различий в качестве данных между моделями.

## Что загружается целиком, а что по требованию

### Целиком для выбранной модели

Загружаются:

- элементы;
- отношения;
- диаграммы;
- связи элементов с диаграммами;
- минимальные metadata fields.

Это нужно, чтобы impact analysis и coverage строились на цельной модели.

### По требованию или как необязательное расширение

Могут загружаться позже:

- расширенные свойства элементов;
- менее значимые metadata blocks;
- дополнительные поля, не влияющие на анализ MVP.

## Жизненный цикл данных в MVP

1. Проверка подключения.
2. Выбор модели.
3. Загрузка source data.
4. Schema-level validation.
5. Graph-level validation.
6. Нормализация в каноническую модель.
7. Построение графа.
8. Предрасчет базовых метрик и quality indicators.
9. Интерактивный просмотр и запуск impact analysis.
10. Экспорт результата.
11. Очистка или сброс session state при смене модели.

## Валидация данных

### Schema-level validation

Проверяет:

- наличие обязательных полей;
- корректность типов;
- возможность безопасно распарсить ресурс.

### Graph-level validation

Проверяет:

- существуют ли ссылки source/target;
- можно ли собрать view-to-element references;
- не нарушаются ли инварианты канонической модели.

## Работа с плохими данными

Система использует принцип `fail soft, warn clearly`.

Это означает:

- по возможности анализ продолжается;
- некорректные или неполные сущности фиксируются как DataIssue;
- пользователь видит предупреждение, если проблема влияет на полноту результата;
- source corruption не должен приводить к silent wrong answer.

## Persisted и transient state

### Transient state MVP

Основной state MVP остается transient:

- загруженная модель;
- граф;
- индексы;
- текущий результат impact analysis;
- текущие фильтры и selection.

### Допустимый persisted state

В MVP допустима persistence только для:

- технической конфигурации подключения;
- логов и диагностических артефактов;
- при необходимости — временного server-side cache.

Не входят в MVP:

- сохраненные analysis snapshots;
- saved views;
- внутренний импорт export files обратно в систему.

## Кэширование

В MVP допускается:

- in-memory session cache;
- optional server-side cache для нормализованной модели и предрасчитанных метрик.

Правило конфликта:

- Architeezy всегда приоритетнее кэша;
- кэш — ускоритель, а не источник истины;
- пользователь должен понимать, когда данные могли устареть.

## Экспортные форматы

### CSV

Обязательные поля CSV должны включать как минимум:

- element id;
- name;
- type;
- layer;
- role in result;
- базовые метрики или quality flags, если релевантно.

### GraphML

GraphML должен содержать:

- nodes подграфа;
- edges подграфа;
- node attributes: `id`, `label`, `type`, `layer`;
- edge attributes: `id`, `type`, `source`, `target`.

Экспорт должен быть воспроизводимым и структурно стабильным.

## Demo datasets

Для разработки и тестирования нужны минимум три набора данных:

- `happy path dataset`;
- `realistic medium dataset`;
- `broken-data dataset`.

Они нужны для стабильной интеграции, проверки correctness и устойчивости к плохим данным.

## Границы интеграций MVP

В MVP не реализуются:

- импорт собственных export-файлов обратно в систему;
- write-back в Architeezy;
- внешние SaaS enrichment-сервисы;
- многоисточниковая интеграция beyond Architeezy.

## Вывод

ArchiLens OSS должен работать с данными как с first-class частью продукта. В первой версии это означает полную загрузку одной модели, строгую нормализацию, явную работу с DataIssue, transient-first runtime state и обязательный export в CSV/GraphML. Именно такой data flow делает impact analysis и coverage не просто красивыми, а воспроизводимыми и доверяемыми.
