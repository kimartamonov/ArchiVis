# 08 System Context and Architecture

## Архитектурная цель MVP

Архитектура ArchiLens OSS должна поддерживать быстрый путь к рабочему MVP, но при этом не запирать продукт в тупиковую front-only конструкцию. Главный принцип — отделение аналитического ядра от пользовательского интерфейса.

## Системный контекст

### Внешние системы

В системный контекст MVP входят:

- `Architeezy API` как основной источник данных;
- `demo datasets` как контролируемый источник для разработки, тестов и демонстрации;
- локальное или self-hosted окружение запуска;
- CI/CD для сборки и проверки проекта.

### Что сознательно не является обязательной частью контекста MVP

- внешние SaaS telemetry-платформы;
- отдельная SSO-интеграция;
- сторонние AI-сервисы;
- внешний persistence platform для хранения аналитических snapshots.

## Архитектурная форма системы

Рекомендуемая форма MVP:

- self-hosted веб-приложение;
- тонкий backend/BFF слой;
- headless graph and insight engine;
- desktop-first UI.

## Почему нужен BFF/backend

Даже для MVP тонкий backend/BFF предпочтительнее чистого frontend с прямым доступом к Architeezy API.

Причины:

- безопаснее работать с credentials и конфиденциальными данными;
- проще управлять конфигурацией подключения;
- появляется единое место для интеграционной диагностики;
- остается возможность позже вынести тяжелые вычисления на server-side без смены публичного UX-контракта.

В MVP backend остается минималистичным и не превращается в отдельную сложную платформу.

## Архитектурные слои

### 1. Connector Layer

Отвечает за:

- обращение к Architeezy API;
- преобразование внешнего контракта в предсказуемый ingress-формат;
- обработку pagination, auth, ошибок источника и частично неполных данных.

### 2. Graph Engine

Отвечает за:

- нормализацию source data;
- построение канонического графа;
- формирование связей между Element, Relationship, View и ElementViewRef;
- инварианты доменной модели.

### 3. Insight Engine

Отвечает за:

- базовые метрики;
- impact traversal;
- quality indicators;
- explainable summary для результатов анализа.

### 4. Analyst UI

Отвечает за:

- выбор модели;
- поиск и навигацию;
- показ графа и таблиц;
- запуск анализа;
- просмотр результата;
- export actions.

## Высокоуровневая структура UI

Основные UI-модули MVP:

- App Shell;
- Source and Model Selection;
- Model Explorer;
- Element Table;
- Element Detail Panel;
- Global Graph View;
- Impact Analysis Panel;
- Coverage and Quality View;
- Export Actions.

## Поток данных между слоями

Базовый поток данных выглядит так:

1. UI запрашивает список моделей через BFF.
2. Пользователь выбирает модель.
3. Connector layer получает source data из Architeezy.
4. Graph engine преобразует source data в канонический граф.
5. Insight engine рассчитывает базовые метрики и quality indicators.
6. UI позволяет выбрать элемент и запустить impact analysis.
7. Insight engine формирует ImpactAnalysisResult и Subgraph.
8. UI отображает результат и инициирует export.

## Где выполняются вычисления

Для MVP выбирается гибридный подход:

- часть аналитики может выполняться в приложении ArchiLens, а не в Architeezy;
- тяжелые вычисления не должны быть жестко привязаны к main UI thread;
- архитектура должна допускать вынос в Web Worker и, при необходимости, в backend execution.

Ключевой принцип:

engine проектируется как headless слой, не зависящий от React/UI.

## Управление состоянием

Состояние приложения должно быть разделено на уровни:

- application state: источник, модель, статусы загрузки;
- navigation state: текущий экран, выбранный элемент;
- analysis state: параметры impact analysis и его результаты;
- view state: фильтры, сортировка, выбранное представление;
- export state: статус формирования и выдачи артефакта.

Для MVP подходит централизованный state container с модульным разделением store по зонам ответственности.

## Синхронные и асинхронные операции

### Асинхронные

- загрузка модели;
- нормализация;
- индексирование;
- impact analysis;
- экспорт;
- потенциально тяжелые пересчеты метрик.

### Синхронные или почти синхронные

- локальная фильтрация уже загруженных данных;
- выбор элемента;
- переключение между режимами представления.

## Развертывание MVP

Базовый deployment model:

- локальный запуск для разработки;
- self-hosted запуск для тестирования и пилота;
- container-friendly packaging.

Это соответствует требованиям конфиденциальности и снижает зависимость от внешней инфраструктуры.

## Архитурные приоритеты MVP

В приоритете:

1. объяснимость результатов;
2. надежность загрузки и нормализации;
3. изоляция интеграционного слоя;
4. возможность пройти end-to-end сценарий;
5. архитектурная готовность к расширению без premature complexity.

## Необратимые решения, которые должны быть зафиксированы

До начала полноценной реализации нужно считать формально значимыми:

- наличие BFF/backend;
- разделение `connector / graph engine / insight engine / UI`;
- read-only интеграцию;
- каноническую доменную модель;
- модель работы с чувствительными данными;
- внутренний контракт между connector и graph engine.

## Вывод

Целевая архитектурная форма ArchiLens OSS — это self-hosted веб-приложение с тонким BFF и headless аналитическим ядром. Такая схема достаточно проста для MVP, но уже поддерживает ключевые требования по безопасности, объяснимости, тестируемости и дальнейшему росту в сторону CLI, batch и extension-сценариев.
