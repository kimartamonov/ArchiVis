# 05 Functional Requirements

## Подход к приоритизации

В проекте используется приоритизация `P0 / P1 / P2`.

- `P0` — обязательное для цельного MVP;
- `P1` — полезное усиление, которое не должно блокировать релиз MVP;
- `P2` — post-MVP или стратегический резерв.

## Трассировка к сценариям

Главный источник требований:

- основной сценарий impact analysis;
- поддерживающий сценарий coverage and hygiene;
- сценарий подготовки evidence-based narrative.

## P0 Functional Requirements

### FR-01 Подключение к источнику данных

Система должна уметь работать с одним настроенным источником Architeezy и получать список доступных моделей.

### FR-02 Выбор одной модели

Система должна позволять пользователю выбрать одну модель для анализа в рамках текущей сессии.

### FR-03 Загрузка модели

Система должна загружать для выбранной модели:

- элементы;
- отношения;
- диаграммы или представления;
- связи между элементами и диаграммными представлениями;
- базовые метаданные, необходимые для отображения и анализа.

### FR-04 Построение канонического графа

После загрузки система должна автоматически нормализовать данные в единый канонический граф модели.

### FR-05 Индексация модели

После построения графа система должна автоматически индексировать элементы для поиска и навигации.

### FR-06 Поиск элемента

Система должна позволять находить элемент минимум по имени.

### FR-07 Навигация к элементу

Система должна позволять выбрать элемент из таблицы, результата поиска или графового представления и открыть его карточку.

### FR-08 Отображение карточки элемента

Система должна отображать карточку элемента с базовой информацией:

- имя;
- тип;
- слой;
- базовые метрики;
- диаграммное присутствие.

### FR-09 Запуск impact analysis

Система должна позволять запускать impact analysis для выбранного элемента.

### FR-10 Управление глубиной анализа

Система должна позволять выбрать глубину impact analysis на 1, 2 или 3 шага.

### FR-11 Результат impact analysis

Система должна показывать:

- список затронутых элементов;
- затронутые слои;
- затронутые диаграммы.

### FR-12 Представление результата impact analysis

Система должна позволять просматривать результат impact analysis как минимум:

- в табличном виде;
- в виде выделенного подграфа.

### FR-13 Переход по связанным элементам

Система должна позволять перейти от результата impact analysis к связанному элементу и продолжить исследование.

### FR-14 Базовые метрики

Система должна рассчитывать базовые метрики модели:

- degree;
- in-degree;
- out-degree;
- diagrams count.

### FR-15 Coverage and hygiene indicators

Система должна выявлять:

- orphan elements;
- low-coverage elements.

### FR-16 Просмотр проблемных элементов

Система должна позволять открыть список проблемных элементов и перейти от него к конкретному элементу.

### FR-17 Explainability для базовых результатов

Система должна объяснять:

- почему элемент попал в impact result;
- почему элемент отмечен как orphan или low-coverage;
- на каких базовых признаках строится краткий risk summary.

### FR-18 Экспорт результата impact analysis

Система должна экспортировать результат impact analysis:

- в CSV;
- в GraphML.

### FR-19 Экспорт coverage result

Система должна экспортировать список проблемных элементов в CSV.

### FR-20 Устойчивость к неидеальным данным

Система должна выполнять best-effort анализ при частично некорректных данных и явно фиксировать ограничения результата.

## P1 Functional Requirements

- поиск по ID элемента;
- фильтрация по диаграмме;
- переключение направления обхода `incoming / outgoing / both`, если не усложняет MVP чрезмерно;
- расширенная risk card;
- richer graph navigation;
- JSON snapshot export;
- более богатые quality filters.

## P2 Functional Requirements

- compare states;
- multi-model analysis;
- saved views;
- shareable links;
- plugin API как продуктовая функция;
- community extension mechanisms;
- alerts on model degradation;
- Cross-Layer X-Ray как отдельный режим;
- Criticality Radar как самостоятельный продуктовый модуль.

## Минимальный набор для цельного MVP

Чтобы MVP считался функционально цельным, должны одновременно работать:

- загрузка одной модели;
- построение графа;
- поиск элемента;
- impact analysis с глубиной 1-3;
- coverage indicators;
- explainability базового результата;
- export.

## Вывод

Функциональное ядро ArchiLens OSS в MVP — это не viewer и не repository companion в широком смысле, а сквозной анализ выбранного элемента и базового качества модели. Все P0-требования должны усиливать именно этот поток и не должны размываться вторичными возможностями.
