# 11 Acceptance Criteria and Test Strategy

## Цель документа

Зафиксировать, как команда поймет, что MVP действительно работает, и какие проверки обязательны до передачи продукта пользователю.

## Главный сценарий приемки

Основной сценарий приемки MVP:

1. Пользователь подключается к источнику данных.
2. Выбирает модель.
3. Система загружает модель и строит граф.
4. Пользователь находит элемент.
5. Запускает impact analysis на выбранную глубину.
6. Получает affected elements, layers и diagrams.
7. Экспортирует результат.

Если этот сценарий не проходит стабильно, MVP не считается принятым.

## Acceptance criteria по областям

### 1. Подключение к Architeezy

Система принята по этому блоку, если:

- соединение устанавливается с настроенным источником;
- пользователь видит понятный статус подключения;
- при ошибке система возвращает диагностируемое сообщение;
- после подключения можно получить список моделей.

### 2. Загрузка модели и построение графа

Система принята по этому блоку, если:

- одна выбранная модель загружается полностью или best-effort образом;
- элементы, отношения и diagram refs собираются в единый канонический граф;
- проблемы данных фиксируются как warnings, а не как silent failures;
- после загрузки доступны поиск и анализ.

### 3. Impact Analyzer

Система принята по этому блоку, если:

- пользователь может выбрать элемент;
- доступен выбор глубины 1-3;
- impact analysis возвращает ожидаемый набор результатов на known dataset;
- видны affected elements, layers и diagrams;
- результат можно просмотреть минимум в двух формах: table and subgraph;
- пользователь может понять, почему результат получился именно таким.

### 4. Coverage and hygiene

Система принята по этому блоку, если:

- orphan elements корректно определяются;
- low-coverage elements корректно определяются;
- от проблемного элемента можно перейти к карточке и impact analysis;
- список проблемных элементов можно выгрузить в CSV.

### 5. Search, filters and navigation

Система принята по этому блоку, если:

- поиск по имени возвращает ожидаемый элемент;
- таблица позволяет фильтрацию по типу и слою;
- можно перейти от результата поиска к элементу;
- можно перейти к связанному элементу из impact result.

### 6. Export

Система принята по этому блоку, если:

- CSV формируется корректно;
- GraphML формируется корректно;
- экспортируемые файлы имеют ожидаемую структуру;
- пользователь понимает, какой именно результат экспортирует.

## Test data strategy

Приемка и тестирование должны использовать три типа datasets:

- `happy-path dataset`;
- `safe real-world or anonymized dataset`;
- `broken-data dataset`.

Это нужно, чтобы одновременно проверить корректность, практическую полезность и устойчивость к плохим данным.

## Known expected outputs

Для тестовых данных должны быть заранее определены:

- expected affected elements для одного или нескольких seed-элементов на глубину 1 и 2;
- expected affected diagrams;
- expected orphan elements;
- expected low-coverage elements.

Без заранее известного expected output нельзя надежно валидировать correctness аналитики.

## Роль ручной экспертной приемки

Автоматических тестов недостаточно. MVP должен проходить ручную валидацию со стороны архитектурного эксперта.

Экспертная приемка проверяет:

- правдоподобность результатов impact analysis;
- понятность summary и explainability;
- практическую полезность для change/review-сценария.

## Минимальный набор обязательных тестов

### Smoke tests

- запуск приложения;
- доступность базового экрана;
- базовая загрузка модели.

### Unit tests

- graph normalization;
- traversal logic;
- metric calculations;
- quality rules.

### Integration tests

- connector behavior;
- data transformation into canonical model;
- error handling on malformed data.

### E2E tests

- главный сценарий impact analysis;
- базовый export scenario.

### Performance smoke

- загрузка модели целевого размера;
- impact analysis на типовом seed-элементе.

## Уровень автоматизации

В первой версии автоматизация должна быть выборочной, но достаточной.

Автоматизировать обязательно:

- core normalization logic;
- impact traversal;
- expected metric outputs;
- basic export schema checks;
- main e2e flow.

Оставить на ручную валидацию допустимо:

- тонкие UX-аспекты graph mode;
- экспертную оценку полезности результата.

## Приоритизация дефектов

### Блокирующие дефекты

- невозможно подключиться к источнику;
- невозможно загрузить модель;
- граф строится некорректно;
- impact analysis падает или выдает явно неверный результат на known dataset;
- export не работает;
- ошибки источника приводят к silent wrong result;
- нарушается безопасность работы с чувствительными данными.

### Допустимые для MVP дефекты

- визуальные шероховатости;
- неполная полировка secondary views;
- ограниченная поддержка нецелевых браузеров;
- неидеальная richness второстепенных фильтров.

## Definition of Done

### Для capability slice

Slice считается done, если:

- реализован наблюдаемый пользовательский или технический результат;
- выполнены acceptance criteria slice;
- есть тесты или validation evidence;
- зафиксированы ограничения.

### Для всего MVP

MVP считается done, если:

- проходит главный end-to-end сценарий;
- выполнены acceptance criteria по подключению, загрузке, анализу, coverage и export;
- есть sample datasets;
- есть manual expert validation;
- есть README, runbook и known limitations;
- performance baseline соблюден.

## Evidence package для готовности MVP

К моменту приемки должны существовать:

- acceptance checklist;
- sample CSV and GraphML exports;
- test report;
- demo script;
- known limitations;
- краткий walkthrough или demo video, если нужен;
- notes from expert validation.

## Вывод

Приемка ArchiLens OSS должна измерять не количество экранов и не completeness фантазии, а способность пройти реальный impact analysis flow на реальных или реалистичных данных с корректным результатом, export-артефактом и достаточным уровнем доверия к аналитике.
