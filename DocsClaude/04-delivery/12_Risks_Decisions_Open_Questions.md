## Принятые решения

### Архитектурные решения (decided)

| ID | Решение | Обоснование | Последствия |
|----|---------|-------------|-------------|
| D-1 | Frontend-only SPA, без backend | Простота деплоя, конфиденциальность данных, нулевая инфраструктура | CORS constraints, нет server-side caching, нет shared state |
| D-2 | React + TypeScript | Зрелая экосистема, сильная типизация, React Flow | Весь UI-стек привязан к React |
| D-3 | React Flow для визуализации графа | Продуктовый UI, node-based интерфейс, хороший DX | Переезд на Cytoscape.js/D3 — дорогой |
| D-4 | Zustand для state management | Лёгкий, без boilerplate, хорошо с React | Пронизывает весь UI |
| D-8 | Graph Engine headless (без UI-зависимостей) | Переиспользуемость в CLI, тестах, будущих UI | Дублирование типов между engine и UI |
| D-9 | ID из Architeezy как primary key | Простота, нет коллизий для single-model MVP | При мульти-модели потребуется namespace |

## Оценка рисков

### Риск 1: API Architeezy не предоставляет нужных данных

| Параметр | Значение |
|----------|----------|
| Вероятность | Средняя (30-50%) |
| Влияние | **Критическое** — без данных нет продукта |
| Категория | Интеграция |
| Ранний сигнал | Spike показывает, что нет endpoint для relationships или diagram membership |
| Мера реагирования | Fallback на demo-only MVP. Искать альтернативные источники (ArchiMate Open Exchange file import) |
| Plan B | MVP работает только на demo dataset. Real integration — Phase 2 |

### Риск 2: Большие графы тормозят UI

| Параметр | Значение |
|----------|----------|
| Вероятность | Низкая-средняя (20-30%) |
| Влияние | Среднее — UX деградирует, но не ломается |
| Категория | Производительность / UX |
| Ранний сигнал | React Flow тормозит при > 300 узлов. Layout > 3 сек |
| Мера реагирования | Ограничить отображаемые узлы (только видимые слои). Упростить layout. Убрать анимации |
| Plan B | Ограничить MVP до 300 элементов. Для больших моделей — только таблица |

### Риск 3: Метрики воспринимаются как «магия»

| Параметр | Значение |
|----------|----------|
| Вероятность | Средняя (30-40%) |
| Влияние | Среднее — потеря доверия, продукт не принимается |
| Категория | UX / доверие |
| Ранний сигнал | Пользователь: «не понимаю, что значит этот score» |
| Мера реагирования | Tooltip к каждой метрике. Показать «почему». Убрать композитные scores, оставить простые (degree, count) |

### Риск 4: CORS блокирует прямые запросы

| Параметр | Значение |
|----------|----------|
| Вероятность | Высокая (60-70%) |
| Влияние | Низкое — решаемо |
| Мера реагирования | Документировать запуск на localhost. Простой CORS proxy (3 строки Express). Попросить Architeezy добавить CORS headers |

### Риск 5: Нет доступа к реальной модели для тестирования

| Параметр | Значение |
|----------|----------|
| Вероятность | Средняя (40%) |
| Влияние | Среднее — невозможно валидировать на реальных данных |
| Мера реагирования | Demo dataset покрывает все тестовые сценарии. Найти open source ArchiMate-модель |

### Риск 6: Bus factor = 1

| Параметр | Значение |
|----------|----------|
| Вероятность | Низкая |
| Влияние | Критическое |
| Мера реагирования | Хорошая документация, чистый код, тесты. Привлечь contributor как можно раньше |

## Следующая волна вопросов после MVP

Если MVP успешен, возникнут:
1. **Масштаб:** Как работать с моделями > 1000 элементов? (Web Workers, агрегация, lazy loading)
2. **Persistence:** Нужно ли сохранять результаты анализа? (Backend, DB)
3. **Collaboration:** Как делиться результатами? (Shareable links, export improvements)
4. **Extensibility:** Как сообщество добавляет свои метрики? (Plugin API)
5. **Multi-source:** Как подключить Archi, TOGAF repos? (Connector SDK)
