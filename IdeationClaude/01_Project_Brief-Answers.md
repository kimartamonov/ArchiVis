# 01_Project_Brief: ответы

## Контекст принятия решений

Ниже — рассуждения и итоговые ответы по каждому вопросу из IdeationPlan. Для каждого вопроса приведена логика выбора, рассмотренные альтернативы и финальное решение.

---

### Кто инициатор проекта?

**Рассмотренные варианты:**
1. Конкурсная заявка для Architeezy — даёт внешний дедлайн, но привязывает к чужим правилам.
2. Внутренний продукт — полный контроль, но нет внешнего валидатора.
3. Open source инициатива — широкий охват, но нужно строить сообщество с нуля.
4. Партнерский showcase — зависимость от партнёра.

**Решение:** Это **независимый open source продукт**, который одновременно является практическим showcase интеграции с Architeezy API. Инициатор — наша команда. Конкурс не является движущим фактором. Мы строим инструмент, который реально полезен архитекторам, работающим с ArchiMate-моделями. Architeezy — первый, но не единственный возможный источник данных.

---

### Кто считается заказчиком?

**Решение:** Заказчик — мы сами как product owners и одновременно целевые пользователи (enterprise/solution архитекторы, которые работают с ArchiMate-моделями). Вторичный заказчик — open source сообщество ArchiMate-практиков. Architeezy не является заказчиком, но является стратегическим партнёром первой интеграции.

---

### Кто принимает продуктовые решения?

**Решение:** Продуктовые решения принимает product owner проекта (в данном случае — основатель инициативы). На этапе MVP это один человек или малая группа (2-3 человека), что позволяет быстро принимать решения без бюрократии. Решения фиксируются в Decision Backlog (документ 13).

---

### Кто принимает архитектурные и технические решения?

**Решение:** Архитектурные решения принимает tech lead / ведущий разработчик. В малой команде это тот же человек, что и product owner. Принцип: архитектурные решения MVP должны быть обратимыми там, где это возможно. Необратимые решения (выбор стека, модель данных, способ интеграции) фиксируются до начала разработки.

---

### Какую главную гипотезу должен подтвердить MVP?

**Рассмотренные варианты:**
1. «Архитекторам нужен impact analysis» — слишком абстрактно.
2. «ArchiMate-модель можно читать как граф зависимостей» — техническая гипотеза, не продуктовая.
3. «Архитектор за 2-3 клика получает ответ на вопрос "что затронет изменение"» — конкретно, проверяемо.
4. «Модель можно автоматически оценить на качество покрытия» — ценно, но вторично.

**Решение:** Главная гипотеза MVP:

> **Архитектор, работающий с ArchiMate-моделью в Architeezy, может за 2-3 клика увидеть полную картину последствий изменения конкретного элемента — включая затронутые элементы, слои и диаграммы — и это реально экономит ему время по сравнению с ручным обходом диаграмм.**

---

### Какой один результат MVP важнее всего?

**Рассмотренные варианты:**
- Доказать полезность impact analysis — самый сильный, напрямую решает боль.
- Доказать ценность quality insights — важно, но вторично.
- Показать силу API Architeezy — побочный эффект, не самоцель.
- Собрать основу для OSS-сообщества — преждевременно для MVP.
- Получить пилотное внедрение — хорошо, но не обязательно на этапе MVP.

**Решение:** Доказать полезность impact analysis на реальной или реалистичной ArchiMate-модели. Это единственный результат, который одновременно впечатляет визуально, решает реальную боль и создаёт основу для всего остального.

---

### Как выглядит ожидаемый outcome после MVP?

**Решение:** Рабочий инструмент, который можно:
1. **Запустить локально** и подключить к реальному экземпляру Architeezy.
2. **Показать в демо** за 5 минут с понятной историей: «выбрал элемент → увидел последствия → увидел диаграммы → выгрузил подграф».
3. **Опубликовать на GitHub** как open source проект с документацией, demo dataset и инструкцией для запуска.

Это не конкурсный артефакт. Это основа для рабочего инструмента, который будет развиваться.

---

### Что будет считаться успехом MVP в измеримом виде?

**Решение:**
1. Пользователь может подключиться к Architeezy API, загрузить модель и увидеть её как граф — за < 30 секунд.
2. По клику на элемент пользователь за < 3 секунды видит affected elements, слои и диаграммы на глубину до 3 шагов.
3. Orphan elements и low-coverage элементы выделяются автоматически.
4. Подграф экспортируется в GraphML и CSV.
5. Демо-прогон по каноническому сценарию проходит успешно на реальной или synthetic-модели от 50 элементов.

---

### Что будет считаться неуспехом или причиной пересмотра идеи?

**Решение:**
- API Architeezy не предоставляет достаточно данных для построения графа связей (нет relationships или нет привязки к диаграммам) — и нет способа обойти это ограничение.
- Impact analysis на реальной модели даёт нерелевантные или непонятные пользователю результаты.
- Архитектор-эксперт при демо говорит: «я и так это вижу в Architeezy, зачем мне ваш инструмент».
- Построение графа на модели > 200 элементов занимает > 10 секунд и делает работу невозможной.

---

### Для какого горизонта планируется первая версия?

**Рассмотренные варианты:**
- 1-2 недели (демо-спринт) — слишком мало для quality MVP.
- 4-6 недель — реалистичный горизонт для малой команды.
- Квартал — слишком долго для проверки гипотезы.

**Решение:** **4-6 недель** от начала разработки до рабочего MVP. Первый демонстрируемый результат — через 2 недели (граф + базовый impact). Полный MVP с coverage-анализом и экспортом — через 4-6 недель.

---

### Какие ограничения уже известны?

**Решение:**
- **Команда:** 1-2 разработчика, product owner совмещает роль.
- **Бюджет:** нулевой денежный бюджет. Только время участников.
- **Сроки:** мягкий дедлайн — 6 недель до первого публичного demo.
- **Доступность экспертов:** нужен хотя бы один человек с опытом ArchiMate для валидации результатов.
- **API Architeezy:** точные возможности API станут понятны после первого spike-этапа.
- **Стек:** ограничение — только open source библиотеки (лицензионно чистые).

---

### Есть ли обязательные внешние ограничения?

**Решение:**
- **Лицензия:** проект должен быть под MIT или Apache 2.0 — широко совместимые open source лицензии.
- **API Architeezy:** работаем в read-only режиме. Никаких записей обратно в репозиторий.
- **Данные:** не храним и не передаём архитектурные модели пользователей на внешние серверы. Всё обрабатывается локально в браузере или на self-hosted сервере пользователя.
- **Корпоративные политики:** не применимы, мы независимый open source проект.

---

### Нужно ли фиксировать географию или тип организаций?

**Решение:** Нет, не нужно. ArchiMate — международный стандарт. Продукт не привязан к географии. Тип организаций для MVP: любая компания, которая использует ArchiMate для описания enterprise-архитектуры и хранит модели в Architeezy. Потенциально — консалтинговые фирмы, крупные enterprise, государственные организации с архитектурным офисом.

---

### Как позиционировать продукт?

**Рассмотренные варианты:**
1. Отдельный OSS-проект — максимальная независимость.
2. Модуль экосистемы Architeezy — зависимость от партнёра, которого у нас пока нет.
3. Reference implementation — ограничивает амбиции.

**Решение:** **Самостоятельный open source проект**, который начинает с интеграции с Architeezy как первого коннектора, но архитектурно не привязан к одному источнику данных. Позиционирование: «analysis layer для ArchiMate-моделей», а не «плагин для Architeezy».

---

### Какой стартовый контекст нужен новому участнику?

**Решение:** Новому участнику нужно понять за 5 минут:

1. **Что мы делаем:** open source веб-инструмент для анализа ArchiMate-моделей — impact analysis, criticality, quality coverage.
2. **Чем это отличается от viewer:** мы не показываем диаграммы. Мы строим единый граф модели и отвечаем на вопрос «что затронет изменение X».
3. **Откуда данные:** из Architeezy по API (read-only). В будущем — из других источников.
4. **Стек:** React + React Flow + elkjs + TypeScript. Графовый анализ — headless layer на TypeScript / Cytoscape.js headless.
5. **Что сейчас важно:** MVP = impact analysis + coverage insights + export. Не редактор, не BI, не замена Architeezy.
6. **Как запустить:** `npm install && npm run dev`, подключить URL Architeezy и token.
