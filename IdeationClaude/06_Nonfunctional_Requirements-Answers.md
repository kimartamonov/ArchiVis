# 06_Nonfunctional_Requirements: ответы

---

### Целевой размер модели для MVP

**Решение:**

| Параметр | Комфортный | Допустимый | Не гарантирован |
|----------|-----------|-----------|-----------------|
| Элементы | до 200 | 200-500 | > 500 |
| Отношения | до 600 | 600-1500 | > 1500 |
| Диаграммы | до 20 | 20-50 | > 50 |

Эти цифры покрывают большинство реальных ArchiMate-моделей в production. Модели крупнейших enterprise (5000+ элементов) — задача Phase 2+ с агрегацией и lazy loading.

---

### Время загрузки модели и построения графа

| Размер модели | Целевое время |
|--------------|---------------|
| До 200 элементов | < 3 сек (включая API-запрос) |
| 200-500 элементов | < 8 сек |
| > 500 элементов | best effort, < 15 сек |

Прогресс-бар должен отображаться для операций > 2 секунд.

---

### Время результата impact analysis

**Решение:** После выбора элемента и глубины — результат должен появиться **за < 1 секунду** для модели до 500 элементов. Это чистый BFS/DFS по уже построенному графу, поэтому операция должна быть мгновенной. Если по каким-то причинам > 1 сек — показывать spinner.

---

### Задержка фильтрации, поиска, навигации

| Операция | Целевое время |
|----------|---------------|
| Текстовый поиск (keystroke → результаты) | < 200 мс |
| Применение фильтра (слой/тип) | < 500 мс |
| Переключение глубины impact (1→2→3) | < 500 мс |
| Pan/zoom графа | 60 fps (без задержки) |
| Переход между экранами | < 300 мс |

### Адаптация под мобильные устройства

**Решение:** **Нет.** Desktop-only в MVP. Граф-based интерфейс на мобильных бесполезен. Таблицы и экспорты теоретически могли бы работать, но затраты на адаптацию не оправданы для MVP.

### Устойчивость при недоступности API

| Ситуация | Поведение |
|----------|-----------|
| API недоступен при подключении | Ошибка с кнопкой retry. Ясное сообщение |
| API отдаёт частичные данные | Загрузка того, что пришло. Warning. Граф строится из доступных данных |
| Timeout при загрузке модели | Ошибка после 30 сек. Кнопка retry |
| API отдаёт ошибку 401/403 | Сообщение «Authentication failed» + redirect на экран подключения |
| Потеря сети во время работы | Работа продолжается на уже загруженных данных. При попытке перезагрузки — ошибка подключения |

Ключевой принцип: после загрузки модели ArchiLens работает **полностью offline**. Все данные — в памяти браузера. Повторные обращения к API не требуются до явной перезагрузки модели.

### Кэширование данных

**Рассуждение:** Варианты:
1. Без кэша — каждый раз загрузка с нуля. Просто, но медленно при повторных визитах.
2. localStorage/IndexedDB кэш — быстрые повторные загрузки, но проблемы с актуальностью.
3. Service Worker кэш — offline support, но сложность.

**Решение:** В MVP — **без persisted кэша**. Данные живут в памяти (in-memory state). При закрытии вкладки данные теряются. При повторном открытии — загрузка заново. Это просто и предсказуемо. Если загрузка слишком медленная — добавление IndexedDB кэша в Phase 2.

Допустимая «протухшесть»: не применимо — данные всегда свежие (загружаются при каждом подключении).

### Требования по безопасности

| Аспект | Решение |
|--------|---------|
| Аутентификация | Пользовательские credentials/token Architeezy. ArchiLens сам не имеет auth |
| Хранение токена | sessionStorage (очищается при закрытии вкладки). Не localStorage |
| Передача данных | HTTPS обязателен для подключения к Architeezy |
| Разграничение доступа | Не применимо — однопользовательское приложение |
| XSS защита | Стандартная React-санитизация. Данные из API не вставляются как innerHTML |
| CORS | Architeezy должен разрешать CORS для домена ArchiLens (или использовать proxy) |

### Работа с конфиденциальными моделями

**Решение:** **Да, это критично.** ArchiLens не передаёт данные модели куда-либо. Всё обрабатывается локально:
- В browser (SPA вариант).
- На self-hosted сервере пользователя (если будет backend).

Никакой telemetry, analytics или передачи данных во внешние сервисы. Это фундаментальное требование для enterprise-пользователей с конфиденциальными архитектурными моделями.

---

### Требования к логированию и диагностике

**Решение для MVP:**
- Console.log для debug (включается через dev mode).
- Error boundary в React — перехват ошибок рендеринга с понятным сообщением.
- Счётчик warnings при загрузке (broken references, unknown types) — отображается в UI.
- Нет внешнего logging service. Нет Sentry/APM в MVP.

---

### Объяснимость метрик как NFR

**Решение:** **Да, это отдельное нефункциональное качество.** Каждая метрика, отображаемая в UI, должна:
1. Иметь **краткое описание** (tooltip) — что означает метрика.
2. Иметь **формулу** (доступна в help/docs) — как она считается.
3. **Не использовать «магические» нормализации** без объяснения.

Пример: `degree = 8` → tooltip: «Total number of direct relationships (incoming + outgoing)».

Это критично для доверия архитекторов к инструменту.

### Доступность (Accessibility)

**Решение:** Минимальный уровень для MVP:
- Контрастные цвета слоёв (различимы при цветовой слепоте — проверить с Coblis).
- Таблицы навигируемы с клавиатуры.
- Alt-text для ключевых элементов UI.
- Графовая визуализация по природе плохо доступна. Таблица + экспорт CSV — альтернативный путь.

Полноценный WCAG 2.1 AA — post-MVP.

---

### Расширяемость в архитектуре MVP

**Решение:** Архитектура MVP должна обеспечить точки расширения, даже если сами плагины не реализованы:

1. **Connector interface** — абстракция для источника данных. Architeezy connector реализует этот интерфейс. Добавление нового коннектора = реализация интерфейса.
2. **Metric interface** — абстракция для метрики. Каждая метрика — отдельный вычислитель с именем, описанием и формулой.
3. **Export interface** — абстракция для экспортёра. GraphML и CSV реализуют интерфейс. Добавление нового формата = реализация интерфейса.

Эти интерфейсы существуют в коде MVP как TypeScript interfaces/types. Runtime plugin system (динамическая загрузка) — post-MVP.

### Сопровождаемость

| Аспект | Решение для MVP |
|--------|----------------|
| Тесты | Unit-тесты для graph engine и метрик. E2E для канонического сценария. Целевое покрытие: > 70% для engine, > 50% для UI |
| Модульность | Чёткое разделение: connector / graph-engine / insight-engine / UI. Каждый — отдельная npm package или директория |
| Документация | README с инструкцией запуска. JSDoc для публичных API. Без избыточного документирования |
| Кодстайл | ESLint + Prettier. Единая конфигурация. Pre-commit hook |
| API-стабильность | Не гарантирована в MVP. Public API помечается как `@beta` |

### Лицензия и юридическая чистота

**Решение:**
- **Лицензия проекта:** MIT — максимально пермиссивная, без ограничений для enterprise.
- **Зависимости:** только MIT, Apache 2.0, BSD-compatible. Никаких GPL в runtime-зависимостях.
- **Проверка:** запустить `license-checker` перед каждым релизом.
- **CLA:** не требуется для MVP. Если появятся внешние контрибьюторы — добавим DCO (Developer Certificate of Origin).
