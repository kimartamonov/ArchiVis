# 02_Scope_and_Boundaries: ответы

## Контекст

Главная задача этого документа — жёстко зафиксировать, что входит в MVP, а что нет. Без этого scope будет расплываться, и вместо работающего инструмента через 6 недель мы получим полуготовый «всё-в-одном».

---

### Поддерживает ли MVP только одну модель за раз или несколько?

**Рассуждение:** Мультимодельный анализ — это Phase 3 фича. Поддержка нескольких моделей одновременно усложняет UI, state management и графовый движок. При этом основной сценарий — impact analysis конкретного элемента — работает в рамках одной модели.

**Решение:** MVP поддерживает **одну модель за раз**. Пользователь выбирает модель при подключении, она загружается полностью. Чтобы работать с другой моделью — переключение через UI (загрузка новой модели с заменой текущей).

---

### Поддерживает ли MVP только Architeezy или нужны другие источники?

**Рассуждение:** Architeezy — единственный источник, для которого мы можем реально построить интеграцию. Добавление импорта из файлов (Open Exchange Format, .archimate) кажется привлекательным, но это отдельный connector + парсер + маппинг. Это удвоит объём интеграционной работы.

**Решение:** MVP поддерживает **только Architeezy API** как источник данных. Архитектура connector layer проектируется так, чтобы в будущем добавить другие коннекторы (ArchiMate Open Exchange, Archi .archimate файлы), но реализация — post-MVP.

---

### Ограничиваем ли MVP read-only режимом?

**Решение:** **Да, строго read-only.** Никаких записей обратно в Architeezy. Никаких мутаций данных в репозитории. ArchiLens — это аналитический слой, а не редактор. Это фундаментальное ограничение, которое упрощает архитектуру, убирает риски повреждения данных и ясно позиционирует продукт.

---

### Нужна ли аутентификация пользователя в самом MVP?

**Рассуждение:** Варианты:
1. Полноценная система авторизации с логинами — overkill для MVP.
2. Настраиваемое подключение (URL + API token Architeezy) — минимально достаточно.
3. Без аутентификации, хардкод подключения — только для dev-режима.

**Решение:** MVP не имеет собственной системы аутентификации. Пользователь указывает URL Architeezy и API token (или credentials) на экране подключения. Токен хранится в session storage браузера. Этого достаточно для MVP.

---

### Какие сценарии входят в MVP как обязательные?

**Решение (P0 — обязательные для MVP):**

| # | Сценарий | Обоснование |
|---|----------|-------------|
| 1 | **Подключение к Architeezy и загрузка модели** | Без этого ничего не работает |
| 2 | **Global Graph** с фильтрами по слою и типу | Основной способ видеть модель как единое целое |
| 3 | **Impact Analyzer** — выбор элемента, просмотр affected elements/layers/diagrams | Главная ценность продукта |
| 4 | **Таблица элементов** с базовыми метриками | Альтернативный способ навигации |
| 5 | **Coverage & Hygiene** — orphan elements, low-coverage | Вторая ключевая ценность |
| 6 | **Export** в GraphML и CSV | Мост к внешним инструментам |

---

### Какие функции сознательно НЕ входят в MVP?

**Решение (явно excluded из MVP):**

| Функция | Почему не в MVP |
|---------|----------------|
| Criticality Radar как отдельный экран | Метрики centrality считаются, но отдельный radar-экран — Phase 2 |
| Cross-Layer X-Ray | Красивая «вау»-фича, но сложная в реализации. Phase 2 |
| PNG/SVG export визуализации | Экспорт данных (GraphML/CSV) важнее. Скриншот пользователь сделает сам |
| JSON snapshot анализа | Nice-to-have, не блокирует MVP |
| Plugin API / Extension API | Архитектурно заложим расширяемость, но plugin system — Phase 2+ |
| Saved views / shareable links | Требует persistence. Post-MVP |
| Compare two states of model | Phase 2-3 |
| Мультимодельный анализ | Phase 3 |
| Уведомления / alerts по деградации | Phase 3 |
| Адаптеры к другим источникам | Post-MVP |
| CLI / batch mode | Post-MVP |
| Мобильный UI | Не нужен для MVP |

---

### Какие метрики обязаны войти в MVP?

**Решение:**

**P0 (обязательные):**
- `degree` (общее число связей)
- `in-degree` (входящие связи)
- `out-degree` (исходящие связи)
- `diagrams_count` (число диаграмм, где элемент присутствует)
- `orphan_flag` (элемент без связей или без диаграмм)

**P1 (желательные, если хватит времени):**
- `impact_radius` (число элементов, затрагиваемых на глубину N шагов)
- `betweenness_centrality` (простая метрика центральности)

**Post-MVP:**
- `eigenvector_centrality`
- `coverage_score` (композитная метрика)
- `risk_heuristics`
- Пользовательские метрики

---

### Насколько глубокой должна быть impact-навигация?

**Решение:** В MVP — **фиксированные 1, 2, 3 шага** с переключателем. Произвольная глубина не нужна: на практике 3 шага уже покрывают большинство полезных сценариев, а дальше граф становится нечитаемым. Ограничение — ради и пользы, и производительности.

---

### Должен ли MVP работать на больших реальных графах?

**Рассуждение:** «Большой» для ArchiMate-модели — это 500-2000 элементов. Для enterprise-моделей бывает и больше, но это скорее исключение для MVP.

**Решение:** Целевой размер для MVP: **до 500 элементов и 1500 отношений**. Модели до 200 элементов должны работать быстро (< 2 сек). Модели 200-500 — с допустимой задержкой (< 5 сек). Модели > 500 — best effort, без гарантий производительности. Это покрывает большинство реальных ArchiMate-моделей.

---

### Какие платформенные границы?

**Решение:** MVP — **только веб, только desktop browser**. Поддерживаемые браузеры: Chrome, Firefox, Edge (последние 2 версии). Мобильный просмотр — не в scope. CLI — не в scope MVP, но архитектура engine должна позволить CLI в будущем.

---

### Demo dataset или реальные модели?

**Решение:** **Оба.** MVP должен:
1. Включать встроенный demo dataset (synthetic ArchiMate-модель, 50-100 элементов) для быстрого старта и демонстраций.
2. Работать с реальными моделями через подключение к Architeezy API.

Demo dataset поставляется в репозитории и может быть загружен без подключения к Architeezy (через локальный mock-connector).

---

### Какие типы экспорта обязательны?

**Решение:**

| Формат | MVP? | Что экспортируется |
|--------|------|-------------------|
| GraphML | **Да** | Выделенный подграф (совместимость с yEd) |
| CSV | **Да** | Таблица элементов с метриками |
| JSON snapshot | Нет | Post-MVP |
| PNG/SVG | Нет | Post-MVP |

---

### Нужна ли настройка quality rules пользователем?

**Решение:** **Нет в MVP.** Правила зашиты в продукт:
- orphan = элемент без связей ИЛИ без присутствия на диаграммах
- low_coverage = элемент с > N связями, но < M диаграмм

Пользовательская настройка правил — Phase 2 (plugin/extension API).

---

### Что считается минимальной полнотой MVP?

**Решение:** MVP считается завершённым, когда можно от начала до конца пройти **канонический сценарий**:

1. Подключиться к Architeezy (или загрузить demo dataset).
2. Увидеть модель как граф с цветовой кодировкой слоёв.
3. Выбрать элемент и увидеть impact на 1-2-3 шага.
4. Увидеть affected диаграммы.
5. Увидеть orphan elements.
6. Экспортировать подграф в GraphML.

Если этот сценарий работает end-to-end — MVP готов.

---

### Что должно быть явно записано как `out of scope`?

**Решение — OUT OF SCOPE для всего проекта на текущем этапе:**

1. Редактирование ArchiMate-модели
2. Создание новых элементов или связей
3. Запись обратно в Architeezy
4. Собственная система управления пользователями
5. Работа с не-ArchiMate моделями (BPMN, UML, etc.)
6. Интеграция с CI/CD пайплайнами
7. Real-time синхронизация с Architeezy
8. AI/ML-анализ архитектуры
9. Генерация рекомендаций по улучшению модели
10. Интеграция с ITSM/CMDB системами
