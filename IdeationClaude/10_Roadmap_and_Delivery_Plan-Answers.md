# 10_Roadmap_and_Delivery_Plan: ответы

---

### Целевая длительность MVP-цикла

**Решение:** **6 недель** от старта разработки до рабочего MVP. С промежуточным checkpoint на неделе 2 (первый демонстрируемый результат).

```
Неделя 0:    Spike + подготовка (API research, project setup)
Недели 1-2:  Ядро (connector + graph engine + базовый UI)
Недели 3-4:  Аналитика (impact analyzer + coverage + таблица)
Недели 5-6:  Экспорт + стабилизация + demo dataset + документация
```

---

### Какая команда предполагается?

**Решение:** **1-2 разработчика** (fullstack, TypeScript/React). Product owner совмещает роль с разработкой. Нет отдельных ролей дизайнера, тестировщика, DevOps.

Реалистичная пропускная способность: 1 человек × 20-30 часов/неделю = 120-180 часов на весь MVP. Это требует жёсткой приоритизации и минимального scope.

### Макро-этапы от идеи до MVP

#### Этап 0: Spike & Setup (Неделя 0, 3-5 дней)

**Цель:** Устранить главную неопределённость — API Architeezy.

**Содержание:**
- Исследовать Swagger UI Architeezy. Получить доступ к API (credentials).
- Сделать ручные запросы. Понять формат данных.
- Подтвердить доступность: elements, relationships, diagrams/views.
- Проверить CORS / auth.
- Настроить проект: React + TypeScript + Vite + ESLint + Prettier.
- Создать базовую структуру папок.
- Подготовить минимальный demo dataset (hardcoded JSON).

**Результат:** Понимание API + работающий skeleton проекта.

**Критерий выхода:** Можем программно получить список элементов и отношений из Architeezy.

---

#### Этап 1: Connector + Graph Engine (Неделя 1)

**Цель:** Загрузить модель и построить граф.

**Содержание:**
- Реализовать Architeezy connector (fetch elements, relationships, diagrams).
- Реализовать нормализацию в NormalizedModel.
- Реализовать Graph Engine: построение графа, adjacency lists, indexes.
- Рассчитать базовые метрики (degree, orphan).
- Unit-тесты для graph engine.

**Результат:** `loadModel(url, token, modelId)` → `AnalysisGraph` с метриками.

**Критерий выхода:** На demo dataset граф строится корректно, метрики совпадают с ручным подсчётом.

---

#### Этап 2: Базовый UI + Global Graph (Неделя 2)

**Цель:** Показать модель визуально. Первый демонстрируемый результат.

**Содержание:**
- Экран подключения (URL + token + Connect).
- Загрузка модели с прогрессом.
- Global Graph на React Flow: узлы = элементы, рёбра = отношения.
- Цветовая кодировка по слоям.
- Автолэйаут (elkjs).
- Zoom/pan.
- Клик по узлу → базовая карточка (имя, тип, degree).

**Результат:** Пользователь подключается → видит граф модели → кликает по узлам.

**Критерий выхода:** Demo dataset отображается как интерактивный граф. Можно показать стейкхолдерам.

---

#### Этап 3: Impact Analyzer (Неделя 3)

**Цель:** Главный сценарий MVP — impact analysis.

**Содержание:**
- BFS/DFS от выбранного элемента.
- Переключатель глубины 1/2/3.
- Отображение affected elements (список + визуальная подсветка на графе).
- Summary по слоям.
- Affected diagrams.
- Карточка элемента с метриками.
- Текстовый поиск по элементам.

**Результат:** Полный impact analysis end-to-end.

**Критерий выхода:** Канонический сценарий (выбрал элемент → увидел impact) работает корректно.

---

#### Этап 4: Таблица + Coverage (Неделя 4)

**Цель:** Второй столп ценности — coverage & hygiene + альтернативная навигация.

**Содержание:**
- Table View (TanStack Table) с сортировкой и фильтрацией.
- Coverage экран: orphan count, orphan list, coverage stats.
- Фильтры по слою и типу (Global Graph + Table).
- Навигация между экранами.
- Переход из таблицы к impact analysis.

**Результат:** Два способа работы с моделью — граф и таблица + coverage insights.

**Критерий выхода:** Пользователь может найти orphan elements и перейти к анализу.

---

#### Этап 5: Экспорт + стабилизация (Неделя 5)

**Цель:** Экспорт + polish + стабильность.

**Содержание:**
- Export GraphML (подграф и полный граф).
- Export CSV (таблица элементов с метриками).
- Финализация demo dataset (реалистичный, с нужными паттернами).
- Bug fixes.
- UI polish: responsive layout, loading states, error messages.
- Проверка на реальной модели Architeezy (если доступна).

**Результат:** Стабильный MVP с экспортом.

**Критерий выхода:** GraphML открывается в yEd. CSV открывается в Excel. Нет критичных багов.

---

#### Этап 6: Документация + Release (Неделя 6)

**Цель:** Готовность к публичному релизу.

**Содержание:**
- README: описание, скриншоты, инструкция запуска.
- CONTRIBUTING.md (если open source).
- LICENSE (MIT).
- Demo script (5 минут, пошаговый).
- Smoke E2E тест.
- GitHub release v0.1.0.
- Deploy demo на GitHub Pages / Vercel (с demo dataset).

**Результат:** Публичный open source проект с документацией и живым демо.

**Критерий выхода:** Новый пользователь может запустить проект по README за < 5 минут.

---

### Критический путь проекта

```
API Research (Spike) → Connector → Graph Engine → Impact Analysis → UI
```

Если API Architeezy не предоставляет нужных данных — проект стопорится на spike. Это главный risk gate. Всё остальное зависит от connector.

**Fallback:** Если API недостаточен, MVP строится только на demo dataset + mock connector. Реальная интеграция — post-MVP.

---

### Внешние и внутренние зависимости

| Зависимость | Тип | Влияние |
|------------|-----|---------|
| Доступ к API Architeezy (credentials) | Внешняя | Блокирует spike. Без этого — только demo dataset |
| Формат API (endpoints, schema) | Внешняя | Определяет connector. Spike должен закрыть |
| CORS policy Architeezy | Внешняя | Может потребовать proxy |
| Тестовая модель в Architeezy | Внешняя | Нужна для проверки на реальных данных |
| React Flow learning curve | Внутренняя | 1-2 дня на освоение |
| elkjs configuration | Внутренняя | Layout может потребовать тюнинга |

---

### Когда можно показать как конкурсный demo?

**Решение:** Не применимо — конкурс не является целью. Но первый впечатляющий demo возможен **после этапа 3 (неделя 3)**: граф + impact analysis работает.

---

### Когда можно дать пилотному пользователю?

**Решение:** **После этапа 5 (неделя 5).** К этому моменту есть: подключение, граф, impact, таблица, coverage, экспорт. Пользователь может получить реальную пользу.

---

### Что убрать первым при нехватке времени?

**Линии сокращения scope (от менее важного к более важному):**

1. ❌ Coverage & Hygiene (этап 4) — можно вырезать, impact analysis остаётся цельным.
2. ❌ CSV экспорт — GraphML достаточно.
3. ❌ Фильтры по типу элемента — фильтр по слою достаточен.
4. ❌ Легенда графа — можно описать в документации.
5. ❌ Прогресс загрузки — заменяем на spinner.

---

### Что нельзя убирать ни при каких условиях?

1. ✅ Подключение к Architeezy (или demo dataset).
2. ✅ Построение графа.
3. ✅ Global Graph с цветовой кодировкой слоёв.
4. ✅ Impact Analysis (выбор элемента → affected elements на 1-2-3 шага).
5. ✅ Экспорт подграфа в GraphML.

Это **minimal viable** набор. Без него MVP теряет смысл.

---

### Нужен ли milestone на explainability?

**Решение:** Не отдельный milestone, а **cross-cutting concern**. На каждом этапе при добавлении метрики — добавляем tooltip с объяснением. Это не занимает отдельный sprint, но важно не забыть.

---

### Roadmap MVP vs Phase 2/3

| Фаза | Горизонт | Содержание |
|------|----------|-----------|
| **MVP** | 6 недель | Connector + Graph + Impact + Table + Coverage + Export |
| **Phase 2** | 2-3 месяца после MVP | Criticality Radar, Cross-Layer X-Ray, Saved Views, Advanced metrics, Plugin API stubs |
| **Phase 3** | 6+ месяцев | Multi-model, Time diff, Alerts, Community extensions, Alternative connectors |

Phase 2 начинается только после подтверждения ценности MVP (real users give positive feedback).

---

### Документация и demo как deliverables?

**Решение:** **Да, это отдельный этап (Этап 6).** Без README и demo script — open source проект мёртв. Обязательные deliverables:
- README.md
- Demo dataset
- Demo script (текстовый, 5 минут)
- Скриншоты/GIF в README

---

### Признаки готовности к post-MVP

1. MVP используется хотя бы 3 реальными пользователями (включая нас).
2. Канонический сценарий работает стабильно на 3+ моделях.
3. Нет critical/blocker багов.
4. Есть хотя бы 1 запрос на фичу из Phase 2 от реального пользователя.
5. Архитектура не требует рефакторинга для добавления Phase 2 фич.
