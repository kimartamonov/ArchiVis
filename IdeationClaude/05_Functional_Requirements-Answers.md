# 05_Functional_Requirements: ответы

---

### Формат приоритизации

**Решение:** Используем **P0 / P1 / P2**:
- **P0** — обязательно для MVP. Без этого MVP нельзя считать завершённым.
- **P1** — желательно для MVP. Включаем, если хватает времени. Усиливает ценность, но без этого MVP всё ещё цельный.
- **P2** — post-MVP. Не реализуем в первой версии, но учитываем в архитектуре.

### Должна ли система подключаться к нескольким экземплярам Architeezy?

**Решение:** **Нет.** MVP работает с одним подключением. Переключение на другой Architeezy — через UI подключения (disconnect + new connection).

---

### Сохранение настроек между сессиями?

**Решение:** **Минимально.** URL и модель запоминаются в localStorage. Токен — в sessionStorage (очищается при закрытии вкладки, по соображениям безопасности). Остальные настройки не сохраняются в MVP.

---

### Какие сущности обязательно загружать?

| Сущность | P0/P1 | Зачем |
|----------|-------|-------|
| Models (список) | P0 | Выбор модели |
| Elements | P0 | Узлы графа |
| Relationships | P0 | Рёбра графа |
| Diagram references (какой элемент на какой диаграмме) | P0 | Coverage, affected diagrams |
| Diagrams (список с именами) | P0 | Отображение affected diagrams |
| Element types / ArchiMate layer | P0 | Цветовая кодировка, фильтры |
| Properties / metadata | P2 | Не нужно для MVP |

---

### Индексация автоматически или по действию?

**Решение:** **Автоматически** после загрузки модели. Пользователь не должен нажимать «Build graph». Загрузка модели = построение графа = расчёт базовых метрик. Всё в одном шаге.

---

### Какие функции поиска обязательны?

**MVP (P0):** поиск по имени (substring match, case-insensitive).
**P1:** фильтр по типу элемента.
**P2:** поиск по ID, тегам, диаграмме.

---

### Какие фильтры обязательны для Global Graph и таблицы?

**MVP (P0):**
- Фильтр по слою ArchiMate (Business, Application, Technology, etc.)
- Фильтр по типу элемента (Application Component, Business Process, etc.)

**P1:**
- Фильтр по degree (> N связей)
- Фильтр orphan only

---

### В какой момент рассчитываются метрики?

**Решение:** Метрики рассчитываются **при построении графа** (сразу после загрузки модели). Это одноразовый расчёт. При изменении фильтров метрики не пересчитываются (фильтры скрывают узлы, но не меняют значения). Impact analysis — это on-demand расчёт при выборе элемента и глубины.

---

### Что система делает с неполной моделью?

| Ситуация | Поведение |
|----------|-----------|
| Отношение ссылается на несуществующий элемент | Warning. Отношение игнорируется в графе. Счётчик broken references |
| Элемент без типа | Отображается с типом «Unknown», серый цвет |
| Дубли элементов (одинаковый ID) | Используется первый. Warning в логе |
| Циклы в графе | Допустимы. Impact analysis работает через BFS с visited set |
| Модель без отношений | Граф отображается (только узлы). Impact analysis недоступен. Warning |
| Модель без диаграмм | Coverage недоступен. Impact analysis работает без affected diagrams |

---

### Функции администрирования в MVP?

**Решение:** **Никаких.** Нет настроек, нет admin panel, нет user management. Единственная «настройка» — параметры подключения к Architeezy на экране connect.

---

### Функции, явно классифицированные как P1/P2

**P1 (желательные для MVP, но не блокирующие):**
- Demo dataset
- Прогресс загрузки
- Направление анализа (in/out/both)
- Centrality highlighting в графе
- Low-coverage detection
- Легенда графа

**P2 (post-MVP):**
- Risk card
- JSON snapshot export
- PNG/SVG export
- Поиск по ID и тегам
- Plugin API
- Saved views
- Cross-Layer X-Ray
- Compare states
