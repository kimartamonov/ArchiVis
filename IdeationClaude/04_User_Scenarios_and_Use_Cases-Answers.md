# 04_User_Scenarios_and_Use_Cases: ответы

---

### Какой один end-to-end сценарий — главный для MVP и demo script?

**Рассуждение:** Нужен сценарий, который за 3-5 минут показывает полную ценность продукта. Он должен быть: конкретным, впечатляющим, проверяемым.

**Решение:** Канонический сценарий MVP — **«Оценка последствий изменения Application Component»**:

> Архитектор узнаёт, что планируется замена application component «Payment Gateway». Он открывает ArchiLens, подключается к модели, находит этот компонент, запускает impact analysis и за 30 секунд видит: какие бизнес-процессы, сервисы, data objects и technology nodes затрагиваются; на каких диаграммах присутствует этот элемент; насколько он критичен. Затем экспортирует подграф в GraphML для yEd и CSV-таблицу для отчёта.

---

### С какого экрана и триггера начинается каждый сценарий?

| Сценарий | Стартовый экран | Триггер |
|----------|----------------|---------|
| Impact Analysis | Model Explorer → поиск элемента | Клик по элементу → кнопка «Analyze Impact» |
| Coverage & Hygiene | Model Explorer → раздел Coverage | Загрузка модели (автоматически вычисляется) |
| Global Graph exploration | Global Graph | Загрузка модели |
| Export | Любой экран с результатами | Кнопка «Export» в toolbar |
| Подключение | Connection screen | Запуск приложения |

---

### Какие предусловия обязательны?

**Решение:**
1. **Architeezy доступен** по сети (или загружен demo dataset).
2. **Пользователь имеет credentials** (URL + API token).
3. **Модель существует** в репозитории и содержит хотя бы 1 элемент, 1 отношение, 1 диаграмму.
4. **Граф построен** — после загрузки модель автоматически нормализуется в граф. Пока граф не построен, анализ недоступен.

---

### Как пользователь находит нужный элемент?

**Рассуждение:** Архитектор знает имя или тип элемента, но не всегда помнит его точное написание. Варианты: текстовый поиск, фильтры, клик по графу, таблица.

**Решение:** Три способа (все доступны в MVP):
1. **Текстовый поиск** по имени элемента (fuzzy search с подсказками) — основной способ.
2. **Клик по узлу в Global Graph** — визуальный способ.
3. **Клик по строке в таблице элементов** — для тех, кто предпочитает таблицы.

Фильтры по типу и слою помогают сузить результаты.

### Пошаговый сценарий Impact Analysis

**Основной поток:**

1. Пользователь открывает ArchiLens. Видит экран подключения.
2. Вводит URL Architeezy и API token. Нажимает «Connect».
3. Система загружает список моделей. Пользователь выбирает модель.
4. Система загружает модель: элементы, отношения, привязки к диаграммам. Строит граф. Показывает прогресс.
5. Пользователь попадает на экран **Model Explorer**. Видит количество элементов, слоёв, диаграмм.
6. Вводит в поиск «Payment» — видит подсказки: «Payment Gateway», «Payment Service», «Payment Process».
7. Выбирает «Payment Gateway» → открывается **Impact Analyzer**.
8. Видит карточку элемента: тип (Application Component), слой (Application), степень (degree: 8, in: 3, out: 5).
9. Видит **affected elements** на 1 шаг: 8 элементов, распределённых по слоям (Business: 2, Application: 4, Technology: 2).
10. Переключает глубину на 2 шага → affected elements расширяется до 19. Видит цветовую кодировку по слоям.
11. Видит **affected diagrams**: список из 4 диаграмм, на которых присутствует «Payment Gateway».
12. Нажимает **Export** → выбирает GraphML → скачивается файл с подграфом 2-шагового impact.
13. Нажимает **Export CSV** → скачивается таблица affected elements с метриками.

**Результат:** Архитектор за < 2 минут получил полную картину последствий изменения, с экспортом для отчёта.

---

### Какие выходы должны появляться по завершении Impact Analysis?

**Решение:**
1. **Список affected elements** — с именем, типом, слоем, расстоянием от исходного элемента.
2. **Визуальный граф affected подграфа** — с цветовой кодировкой по слоям.
3. **Summary по слоям** — сколько элементов затронуто на каждом слое.
4. **Список affected diagrams** — диаграммы, на которых присутствует исходный элемент.
5. **Возможность экспорта** — GraphML, CSV.

**Не в MVP:** risk card (отдельная оценка риска с score) — это P1, если хватит времени.

### Пошаговый сценарий Model Coverage & Hygiene

**Основной поток:**

1. Пользователь загрузил модель (шаги 1-5 из Impact Analysis).
2. Переходит на экран **Coverage & Quality** (вкладка в навигации).
3. Видит **общие метрики модели:**
   - Всего элементов: 247
   - Элементов без связей (orphans): 18 (7.3%)
   - Элементов без диаграмм: 34 (13.8%)
   - Элементов с > 5 связями, но < 2 диаграмм: 12
4. Видит **список orphan elements** — таблица с возможностью сортировки.
5. Видит **heatmap по слоям** — какой слой покрыт хуже всего.
6. Кликает на orphan element → переходит в его карточку (Model Explorer).
7. Может экспортировать список проблемных элементов в CSV.

**Результат:** Владелец репозитория за 1 минуту получает картину качества модели и список элементов для cleanup.

---

### Нужен ли отдельный сценарий для Criticality Radar в MVP?

**Решение:** **Нет, как отдельный экран — не в MVP.** Однако элементы с высоким degree и betweenness_centrality подсвечиваются в таблице и графе (визуально выделены размером узла или цветом). Это даёт эффект «радара» без отдельного экрана. Полноценный Criticality Radar — Phase 2.

### Сценарий подготовки evidence-based narrative для стейкхолдеров

**Основной поток:**

1. Архитектор проводит impact analysis (сценарий выше).
2. Экспортирует подграф в GraphML → открывает в yEd → делает красивый layout → сохраняет PNG.
3. Экспортирует таблицу affected elements в CSV → открывает в Excel → форматирует для отчёта.
4. Вставляет в презентацию: «Изменение Payment Gateway затрагивает 19 элементов на 3 слоях, включая 2 критичных бизнес-процесса».

**Решение:** В MVP ArchiLens даёт raw data (GraphML + CSV). Красивое оформление — на стороне пользователя в привычных инструментах. Это реалистичный подход для MVP.

### Какие сценарии поддерживают экспорт, а какие только просмотр?

| Сценарий | Экспорт | Только просмотр |
|----------|---------|----------------|
| Impact Analysis | GraphML подграфа, CSV affected elements | — |
| Coverage & Hygiene | CSV orphan/low-coverage elements | — |
| Global Graph | GraphML полного графа | — |
| Table View | CSV всех элементов с метриками | — |
| Model Explorer / поиск | — | Только просмотр |

**Решение:** Экспорт доступен из всех основных результатов анализа. Поиск и навигация — только для просмотра.

### Какие ошибки и отклонения нужно описать?

| Отклонение | Поведение системы |
|-----------|------------------|
| Элемент не найден по поиску | Сообщение «No elements found. Try a different search term.» |
| API Architeezy не отвечает | Сообщение «Cannot connect to Architeezy. Check URL and credentials.» + retry кнопка |
| Модель слишком большая (> 500 элементов) | Warning: «Large model detected. Performance may be reduced.» Граф строится, но с предупреждением |
| Граф не строится (нет отношений) | Сообщение «No relationships found in this model. Impact analysis requires relationships.» Таблица и coverage доступны |
| Модель пустая | «No elements found in this model.» |
| Токен невалидный / expired | «Authentication failed. Please check your API token.» |
| Данные содержат битые ссылки | Warning в UI: «N broken references found.» Битые ссылки игнорируются при анализе, элементы с битыми ссылками помечаются |

### Какие сценарии на реальной модели, какие на demo?

**Решение:** Все сценарии должны работать на обоих типах данных. Demo dataset проектируется так, чтобы содержать:
- Хотя бы 1 «хаб» (элемент с 8+ связями) — для впечатляющего impact analysis.
- Хотя бы 5-10 orphan elements — для демонстрации coverage.
- Элементы на всех 3 слоях ArchiMate — для демонстрации cross-layer анализа.
- 5-10 диаграмм с разной степенью покрытия.

---

### Какие сценарии явно исключены из MVP?

1. **Cross-Layer X-Ray** — вертикальная визуализация от business к technology.
2. **Compare two states** — сравнение версий модели.
3. **Saved analysis** — сохранение и повторное открытие результатов.
4. **Shared links** — отправка URL с результатом анализа коллеге.
5. **Batch analysis** — анализ всех элементов разом с автоматическим отчётом.
6. **Custom quality rules** — пользовательские правила для coverage check.
